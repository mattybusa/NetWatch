FEATURE SUMMARY: Dedicated Service Account (netwatch-svc)
==========================================================
Type: Feature
Date: 2026-03-13
Sessions: Session 10
Final outcome: Complete — v3.3.25


BACKGROUND
----------
Both netwatch-monitor and netwatch-web were running as mboynton, the primary Pi user
with full sudo privileges. Goal: run both services as a dedicated no-login system
account (netwatch-svc) with the minimum privileges required. This narrows the blast
radius if a service is ever compromised — an attacker gets a locked-down account with
no shell, no home directory, and only specific sudoers entries.


WHAT WAS TRIED AND FAILED
--------------------------
Attempt 1 — Wrong approach: move netwatch dir to /home/netwatch-svc/
A first package was built that moved ~/netwatch to /home/netwatch-svc/netwatch and
updated all shell scripts to use absolute /home/netwatch-svc/ paths. This was
fundamentally wrong: it created a home directory requirement for the service account,
defeating the security benefit of --no-create-home. The account was created with
--create-home (wrong), the user caught the error at Step 3 of manual instructions, and
the account was deleted with sudo userdel -r netwatch-svc. That package was discarded.

Attempt 2 — Package installed but patcher reported "syntax error":
The correct rebuild (v3.3.25b) was installed via Package Installer and failed with:
  "syntax error in 'main.py': [Errno 13] Permission denied:
   '/tmp/__pycache__/tmptmiqxbw3.cpython-313.pyc'"
The error message says "syntax error" but the real cause was a permissions problem on
/tmp/__pycache__/. The directory had accumulated 11MB of temp .pyc files from every
previous package install (all owned by mboynton), and netwatch-svc could not write
there. Fixed with sudo chmod 777 /tmp/__pycache__, then install succeeded.

Attempt 3 — Monitor crashed immediately after manual restart before package installed:
After completing manual Steps 1-9 and restarting services, the monitor crashed with:
  PermissionError: [Errno 13] Permission denied: '/nonexistent'
  at: os.makedirs(os.path.dirname(LOG_PATH), exist_ok=True)
The monitor was restarted before the package was installed. The old main.py still on
disk used expanduser("~/netwatch/logs/netwatch.log"). For netwatch-svc with home-dir
/nonexistent, this resolves to /nonexistent/netwatch/logs/ which does not exist and
cannot be created. Fix: stop the monitor, install the package first (which delivers
the fixed main.py), then restart the monitor.

Web service failed with status=200/CHDIR:
netwatch-web could not start because systemd (running as netwatch-svc) could not
chdir into WorkingDirectory=/home/mboynton/netwatch. Root cause: /home/mboynton was
700 (drwx------). netwatch-svc needs execute permission on the parent directory to
traverse into it. Fixed with sudo chmod 751 /home/mboynton. The 751 gives others
execute-only (traverse) without exposing directory contents.


WHAT WORKED
-----------
Final approach: netwatch-svc with no home directory, files stay at ~/netwatch/

SERVICE ACCOUNT CREATION (manual, before package):
  sudo useradd --system --shell /usr/sbin/nologin \
    --home-dir /nonexistent --no-create-home netwatch-svc
  sudo usermod -aG netwatch-svc mboynton

FILE OWNERSHIP TRANSFER (manual, before package):
  sudo chown -R netwatch-svc:netwatch-svc ~/netwatch
  sudo chmod -R 750 ~/netwatch
  sudo chmod 751 /home/mboynton   ← required for traversal

SUDOERS (manual, before package, written to /etc/sudoers.d/netwatch-svc, chmod 440):
  netwatch-svc ALL=(ALL) NOPASSWD: /usr/bin/systemctl restart netwatch-web.service
  netwatch-svc ALL=(ALL) NOPASSWD: /usr/bin/systemctl restart netwatch-monitor.service
  netwatch-svc ALL=(ALL) NOPASSWD: /usr/bin/systemctl stop netwatch-web.service
  netwatch-svc ALL=(ALL) NOPASSWD: /usr/bin/systemctl stop netwatch-monitor.service
  netwatch-svc ALL=(ALL) NOPASSWD: /usr/bin/systemctl start netwatch-web.service
  netwatch-svc ALL=(ALL) NOPASSWD: /usr/bin/systemctl start netwatch-monitor.service
  netwatch-svc ALL=(ALL) NOPASSWD: /usr/bin/apt-get update
  netwatch-svc ALL=(ALL) NOPASSWD: /usr/bin/apt-get upgrade -y
  netwatch-svc ALL=(ALL) NOPASSWD: /usr/bin/apt-get upgrade --dry-run

SERVICE UNIT FILES (manual, sed commands):
  sudo sed -i 's|User=mboynton|User=netwatch-svc|' \
    /etc/systemd/system/netwatch-monitor.service
  sudo sed -i 's|User=mboynton|User=netwatch-svc|' \
    /etc/systemd/system/netwatch-web.service
  sudo systemctl daemon-reload
  WorkingDirectory and ExecStart paths stay as /home/mboynton/netwatch/ — correct,
  since the files were not moved.

CRONTAB: Existing mboynton crontab entries for backup/pkg_update migrated to
  netwatch-svc's crontab (sudo crontab -u netwatch-svc -e) and removed from mboynton's.
  webapp.py calls crontab -l as the running user — after transition it reads
  netwatch-svc's crontab, which is the correct behavior.

KNOWN LIMITATIONS / DEFERRED ITEMS
------------------------------------
- /tmp/__pycache__/ will accumulate temp .pyc files from patcher again. Real fix:
  patcher.py should use tempfile.mkstemp() and clean up after syntax check. Tracked
  as backlog item for a future session.

- install.sh now handles service account creation for fresh installs. It does NOT
  handle the upgrade case (existing mboynton-owned installs) — that required manual
  steps this session. If another Pi is set up from scratch using install.sh, the
  service account steps are automated. Upgrading an existing install still requires
  the manual procedure documented here.
