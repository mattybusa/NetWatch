ARC: DB Export Fix (v3.3.31)
================================================================================

BACKGROUND
----------
DB backup button in Admin UI was returning 404. Feature had presumably worked at some
point but was broken, likely since the gunicorn migration or service account work.

WHAT WAS TRIED AND FAILED
--------------------------
First package (v3.3.31) installed but 404 persisted. This appeared to be a code
failure but was actually a deployment failure — the web service never restarted after
install due to the sudoers mismatch bug (see below). Old webapp.py was still running.
Took checking journalctl to discover the restart was blocked.

WHAT WORKED
-----------
Root cause: admin_backup_db() in webapp.py had no @app.route decorator, no
@auth.login_required, and no @auth.requires_permission decorator. The function existed
and contained correct implementation logic, but was never registered as a Flask route.
Flask silently ignored it; any request to /api/admin/backup_db got a 404.

Fix: added three lines above the function definition:
  @app.route("/api/admin/backup_db")
  @auth.login_required
  @auth.requires_permission("manage_admin")

Confirmed correct URL by searching admin.html in the zip:
  <a href="/api/admin/backup_db" ...>Download Database Backup (.db)</a>

Shipped in webapp.py in v3.3.31 package (combined with wsgi.py logging fix).

KNOWN LIMITATIONS / DEFERRED ITEMS
------------------------------------
None.

ARC: Web Service Logging Fix (v3.3.31)
================================================================================

BACKGROUND
----------
Discovered while diagnosing the snapshot pruning issue. Added debug logging to
patcher.py but saw no output in journalctl. Investigation revealed wsgi.py had no
logging.basicConfig() call — all log output from the web service was silently dropped.

WHAT WAS TRIED AND FAILED
--------------------------
None — once identified the fix was obvious.

WHAT WORKED
-----------
Root cause: wsgi.py (gunicorn entry point) had no logging configuration. main.py
(monitor entry point) had logging.basicConfig() with FileHandler + StreamHandler.
wsgi.py had neither. All log.info(), log.error() etc. from patcher.py, webapp.py,
and every other module imported by the web service were silently dropped since
the gunicorn migration.

Fix: added logging.basicConfig() to wsgi.py matching main.py pattern exactly:
  - level=logging.INFO
  - format: "%(asctime)s [%(levelname)s] %(name)s — %(message)s"
  - handlers: FileHandler(LOG_PATH) + StreamHandler(sys.stdout)
  - LOG_PATH: same logs/netwatch.log file as monitor service

Both services now write to the same log file and both appear in journalctl.
Shipped in wsgi.py in v3.3.31 package.

KNOWN LIMITATIONS / DEFERRED ITEMS
------------------------------------
None.

ARC: Snapshot Pruning Fix (v3.3.32)
================================================================================

BACKGROUND
----------
User noticed 44 snapshots in ~/netwatch/snapshots/ with SNAPSHOT_RETENTION=10.
Pruning had never worked — snapshots had been accumulating since v3.3.0 (March 10).

WHAT WAS TRIED AND FAILED
--------------------------
Multiple dead ends before finding root cause:

1. Suspected ownership mismatch — confirmed all snapshots owned by netwatch-svc.
   Not the problem.

2. Suspected permissions on snapshots/ directory — confirmed drwxrwx--- owned by
   netwatch-svc. netwatch-svc has full write access. Not the problem.

3. Simulated pruning logic in Python — confirmed the algorithm is correct.
   Not the problem.

4. Tested interactive deletion as mboynton — failed with Permission denied.
   Expected (snapshots/ has no group write for mboynton). Not a bug.

5. Tested interactive deletion as netwatch-svc — succeeded. Not the problem.

6. Checked for early returns in _take_snapshot before pruning block — none found.

7. Added debug logging to patcher pruning block — no output in journalctl.
   Led to discovery of wsgi.py logging bug (fixed first in v3.3.31).

8. After wsgi.py fix, reran package install — pruning log lines appeared.
   Revealed: "name 'config' is not defined" — the actual root cause.

WHAT WORKED
-----------
Root cause: patcher.py used getattr(config, "SNAPSHOT_RETENTION", 10) in the pruning
block without importing config at the module level. When patcher is imported by
webapp.py, config happens to be in the shared namespace and the call works
incidentally. But this reliance is not guaranteed and had been failing silently since
deployment — masked by the original bare "except Exception: pass".

Fix: added "import config" to patcher.py imports section alongside other imports.

Debug logging added in v3.3.30-v3.3.32 was cleaned up in v3.3.33. Final pruning
block retains permanent error logging (failures now visible in journalctl) but
removes the verbose info lines used for diagnosis.

On first install after fix, pruning ran and deleted 34 old snapshots, leaving 10.
Confirmed by user.

Files changed:
- patcher.py: added "import config" to imports; replaced bare except+pass with
  logged error handling in pruning block.

KNOWN LIMITATIONS / DEFERRED ITEMS
------------------------------------
- rollback.sh does not prune snapshots after restoring. Pruning only runs when
  patcher takes a new snapshot. If many rollbacks are performed without package
  installs, snapshots can accumulate. This is a known gap, not currently a backlog
  item — low priority since rollbacks are rare.
