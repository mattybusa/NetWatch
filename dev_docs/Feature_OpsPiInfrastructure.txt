BACKGROUND
----------
The original gitea-pi (Pi 3B+) was set up in Session 11 to run Gitea only.
Session 15 decided to add Prometheus + Loki + Grafana + Node Exporter to
that same machine to complete the observability stack. The Pi 3B+ has 1GB
RAM which is borderline for running Gitea + full observability stack
simultaneously (estimated 500-650MB for all four services).

Decision made in Session 16: use the Pi 4 8GB that was reserved for a future
project. The observability stack is the right use for it -- it will serve
all future projects as the central monitoring/logging infrastructure.

Simultaneously: gitea-pi renamed to ops-pi. The name "gitea-pi" was already
inaccurate after one day. ops-pi is broad enough to remain accurate as more
services are added.

WHAT WAS TRIED AND FAILED
--------------------------

SD card swap (Pi 3B+ -> Pi 4):
Considered but rejected. Booting a 3B+ SD card on a Pi 4 is unreliable due
to model-specific firmware in the boot partition. Since a full new stack was
being installed anyway, a fresh install was the correct choice.

Gitea 1.22.6:
Downloaded and installed initially. Crashed immediately with:
  "Migration Error: Your database (migration version: 323) is for a newer
  Gitea, you can not use the newer database for this old Gitea release (299)."
The SSD database was created by Gitea 1.25.5 (the version running on the 3B+).
Migration version 323 corresponds to Gitea 1.25.x. Fixed by downloading
Gitea 1.25.5 instead.

Gitea app.ini permission denied:
After installing 1.25.5, Gitea crashed with:
  "Error saving internal token: failed to save app.ini: permission denied"
Gitea needs write access to app.ini on first startup to generate and save its
internal token. app.ini was owned root:gitea with 640 permissions -- gitea
user could read but not write. Fixed by temporarily setting 660 before first
start. After successful start, locked back to 640.

git not installed:
After fixing the permission issue, Gitea crashed with:
  "git not found: exec: git: executable file not found in $PATH"
Fresh Pi OS does not include git. Fixed with: sudo apt install git -y

Loki 3.5.1 config -- missing delete_request_store:
Initial Loki config had retention_enabled: true in the compactor block but
did not set delete_request_store. Loki 3.x requires this field when retention
is enabled. Error on startup:
  "invalid compactor config: compactor.delete-request-store should be
  configured when retention is enabled"
Fixed by adding: delete_request_store: filesystem

Prometheus 3.5.1 -- missing consoles/ directories:
After extracting the Prometheus tarball, attempted to copy consoles/ and
console_libraries/ directories to /etc/prometheus/. These directories do not
exist in Prometheus 3.x releases -- they were removed. The cp commands
failed with "cannot stat". Not needed for our setup; omitted.

SCP host key warning (NetWatch Pi -> ops-pi):
When SCP-ing the new TLS cert from NetWatch Pi to ops-pi, the NetWatch Pi's
known_hosts still had the old 3B+ host key for 192.168.100.230. Fixed with:
  ssh-keygen -f '/home/mboynton/.ssh/known_hosts' -R '192.168.100.230'

TLS certificate for ops-pi.home:
Generated on NetWatch Pi using existing LAN CA:
  cd ~
  openssl genrsa -out ops-pi.key 2048
  openssl req -new -key ops-pi.key -out ops-pi.csr \
    -subj "/C=US/ST=Home/L=HomeNetwork/O=OPSPi/CN=ops-pi.home"
  cat > ops-pi-ext.cnf << EOF
  subjectAltName=DNS:ops-pi.home,DNS:ops-pi,IP:192.168.100.230
  EOF
  sudo chmod 664 ~/netwatch/certs/ca/netwatch-ca.srl
  openssl x509 -req -in ~/ops-pi.csr \
    -CA ~/netwatch/certs/ca/netwatch-ca.crt \
    -CAkey ~/netwatch/certs/ca/netwatch-ca.key \
    -CAcreateserial -out ~/ops-pi.crt -days 3650 \
    -extfile ~/ops-pi-ext.cnf
Note: openssl commands must be run from ~ (home dir), not from
~/netwatch/certs/ca/ -- that directory is owned by netwatch-svc and
mboynton cannot write there.

Cert/key placed at /etc/gitea/certs/gitea.crt and gitea.key.
Owned root:gitea, 640 permissions.
