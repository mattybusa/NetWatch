  incremental patching introduced duplicate Flask route decorators. Flask silently ignores
  the second definition, causing old route behavior to persist. Fixed by cleaning duplicate
  route blocks from webapp.py. Rule established: after patching webapp.py, always check
  for duplicate @app.route decorators.

- Redirect loop on guest login (v2.6): After registering as a guest, the login redirect

ADDITIONAL FINDINGS — Feature_PatcherSystem.txt
------------------------------------------------

DUPLICATE ROUTE PREFLIGHT CHECK (v2.4):
After the duplicate @app.route('/preferences') issue (see original summary and Gotchas),
the patcher was updated to perform a preflight check on webapp.py specifically:
Before applying any package that includes webapp.py, patcher.py now scans the
replacement file for duplicate @app.route decorators and refuses to apply if any are
found, reporting the duplicate route names. This prevents deploying a broken webapp.py.
Rule: duplicate route detection runs in apply_package() on any .py file that uses
@app.route.

API RESPONSE FORMAT — GET /api/auth/roles (v2.4):
The roles API route initially returned {"roles": [...]} (a wrapper object).
The admin_users.html JavaScript called roles.map() directly on the response —
it expected a plain array, not a wrapper object. This caused "roles.map is not a
function" TypeError. Fixed by changing the API to return a plain array: jsonify([...])
instead of jsonify({"roles": [...]}). Rule: check whether template JS expects a
wrapped or plain response format when adding new API endpoints.

ADDITIONAL FINDINGS — Bug_UTCTimestampMigration.txt
----------------------------------------------------

ISO 8601 'T' SEPARATOR BUG (additional detail):
After the UTC migration, the 1H chart still showed wrong record counts. The root cause
was that datetime.utcnow().isoformat() stores timestamps as "2026-03-01T09:24:11"
(with a 'T' separator), but SQLite's datetime('now') returns "2026-03-01 09:24:11"
(with a space separator). SQLite string comparison sees 'T' (ASCII 84) as greater than
space (ASCII 32), so nearly all records passed the >= check regardless of their actual
timestamp. Fix: normalize stored timestamps to use space separator instead of 'T', or
replace() in the query. The fix chosen was to use replace() in all get_health_history()
and get_speedtest_history() queries: WHERE timestamp >= datetime('now', '-N hours')
works because we additionally convert stored timestamps to space-separated format.
This is a separate issue from the original UTC migration and was fixed in the same session.

TIMEZONE DISPLAY FIX — 'Z' SUFFIX:
After UTC migration, metric chart timestamps displayed 5 hours ahead of local time.
Cause: JavaScript's new Date("2026-03-01T09:24:11") treats a timestamp without
timezone suffix as local time, not UTC. Fix: append 'Z' to all timestamps when
sending them from Flask to the browser: timestamp + 'Z'. JavaScript then parses as
UTC and toLocaleTimeString() displays in the browser's local timezone automatically.
Applied in all JSON responses that include timestamp fields.

SPEEDTEST CHART RANGE NOT UPDATING (v2.8):
Separate from the UTC migration: when the user changed the time range selector
(1H/6H/24H), the speedtest charts were not re-fetching data. The chart data was
fetched once on page load and the range selector change event was not wired to
re-fetch. Fix: add a change event listener on the range selector that calls
loadSpeedtestData() with the new range parameter.
