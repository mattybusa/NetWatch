NETWATCH EXPANDED SUMMARY — FEATURE: ROLES AND PERMISSIONS SYSTEM
==================================================================
Name: Role System Hardening — is_system, Delete Protection, Alert Defaults
Type: Feature
Date: 2026-03-06 (v3.2.26–v3.2.28)
Sessions: Session 1.3
Final outcome: Complete


BACKGROUND
----------
The built-in roles (Admin, Monitor, Operator) had no protection against deletion.
The roles UI showed them with a "Custom" tag incorrectly. Alert defaults were global
rather than per-role. These were all addressed in the v3.2.26–v3.2.28 arc.


WHAT WAS TRIED AND FAILED
--------------------------
Version suffix 3.2.24b comparison bug:
The version string "3.2.24b" was treated by the packaging library as a pre-release
of "3.2.24" (i.e., older). The patcher's version order check flagged 3.2.24b as out
of order. Rule established: no letter suffixes in version numbers going forward.
Use 3.2.24.1 or 3.2.25 style. The version scheme was normalized.

reportlab not installed on Pi:
The initial changelog PDF implementation used reportlab. When the package was installed
on the Pi, PDF generation failed because reportlab was not in requirements.txt and was
not pre-installed. Solution: replaced reportlab with a pure-HTML approach — changelog
entries rendered as styled HTML files, displayed in an iframe in the admin UI.
PDF generation removed as a separate step.


WHAT WORKED
-----------
IS_SYSTEM FLAG (v3.2.26):
roles table: added is_system BOOLEAN column. Populated via migration:
  Admin, Monitor, Operator → is_system = True
  All others → is_system = False

UI behavior:
- is_system = True: no delete button shown, no "Custom" tag, role name not editable
- is_system = False: delete button shown (with reassignment dialog), "Custom" tag shown

ROLE DELETE WITH REASSIGNMENT DIALOG (v3.2.26–v3.2.27):
When deleting a custom role that has users assigned:
1. Dialog: "N users are assigned to [RoleName]. Choose a role to reassign them to:"
2. Dropdown of all other roles
3. Confirm → reassign all users, then delete role
Roles with zero users: delete immediately with confirmation.

ROLE ALERT DEFAULTS (v3.2.21):
Per-role alert type defaults stored in role_alert_defaults table:
  role_id, alert_type, channel (email|sms), enabled (boolean)
When a new account is created, their subscriber record is initialized from their
role's defaults. Subscriber overrides always win over role defaults (defaults are
just the starting point, not a lock).

UI location: Users & Roles page, collapsible "Alert Defaults" section below role
cards. Shows a matrix: rows = alert types, columns = roles, cells = email/SMS toggles.
Admin can set defaults for each role from this view.

ROLE ALERT DEFAULTS PANEL REFRESH ON DELETE (v3.2.31):
When a role is deleted, the role defaults panel refreshes immediately without page reload.
deleteRole() and confirmRoleDelete() both call loadRoleDefaults() after successful
deletion, but only if the defaults panel is currently open (checked by panel visibility).

CUSTOM ROLE ROW CAP (v3.2.27–v3.2.28):
Role cards in the Users & Roles page capped at 5 per row (user requested).
When more than 5 roles exist, they wrap to a new row. The overflow centering behavior
was fixed to align consistently regardless of count.

CREATE USER BUG (v3.2.29):
must_change_pass was being silently ignored on user creation — always set to False
regardless of the checkbox state. Fixed: the checkbox value is explicitly read and
passed to create_user().
JS hoisting bug: the "Create User" button's click handler was bound before the modal
HTML existed in the DOM (called at script parse time, not DOMContentLoaded). Fixed
by moving handler binding to DOMContentLoaded.

KNOWN LIMITATIONS / DEFERRED ITEMS
------------------------------------
- Role names for system roles (Admin, Monitor, Operator) cannot be renamed via UI.
  They are hardcoded as system roles. A future feature could allow renaming system
  roles while keeping the is_system protection.
- Role alert defaults apply at account creation time. Changing a role's defaults does
  not retroactively update existing users' subscriptions.
