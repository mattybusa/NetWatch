1.2 NETWATCH DIRECTORY & PERMISSIONS
--------------------------------------

[GAP] Parent directory traversal: sudo chmod 751 /home/mboynton
  WHY: netwatch-svc runs with WorkingDirectory=/home/mboynton/netwatch. The
  default /home/mboynton is 700 (owner-only). The service account needs execute
  permission on the parent to traverse into it. Without this, service fails at
  startup with status=200/CHDIR (Permission denied).
  SESSION: 10 | GOTCHA: SERVICE ACCOUNT HOME DIRECTORY TRAVERSAL

1.3 SERVICE ACCOUNT (netwatch-svc)
------------------------------------
[IN INSTALL.SH] Account created:
  sudo useradd --system --shell /usr/sbin/nologin
    --home-dir /nonexistent --no-create-home netwatch-svc
[IN INSTALL.SH] Installing user added to group:
  sudo usermod -aG netwatch-svc mboynton
[IN INSTALL.SH] netwatch-svc added to gpio group:
  sudo usermod -aG gpio netwatch-svc
  WHY: RPi.GPIO requires /dev/mem access for GPIO output. Without gpio group
  membership, GPIO imports succeed (GPIO_AVAILABLE=True, no simulation warning)
  but all GPIO.output() calls silently fail. Relays do not fire. Discovered
  Session 40 when relay worked from CLI (as mboynton) but not from NetWatch
  (as netwatch-svc). Fix: usermod -a -G gpio netwatch-svc + restart monitor.
  NOTE: lgpio (used by button.py) uses /dev/gpiochip and does NOT need gpio
  group membership — but relay.py still uses RPi.GPIO and does need it.
[IN INSTALL.SH] Sudoers written to /etc/sudoers.d/netwatch-svc (chmod 440):
  netwatch-svc ALL=(ALL) NOPASSWD: /usr/bin/systemctl restart netwatch-web.service
  netwatch-svc ALL=(ALL) NOPASSWD: /usr/bin/systemctl restart netwatch-monitor.service
  netwatch-svc ALL=(ALL) NOPASSWD: /usr/bin/systemctl stop netwatch-web.service
  netwatch-svc ALL=(ALL) NOPASSWD: /usr/bin/systemctl stop netwatch-monitor.service
  netwatch-svc ALL=(ALL) NOPASSWD: /usr/bin/systemctl start netwatch-web.service
  netwatch-svc ALL=(ALL) NOPASSWD: /usr/bin/systemctl start netwatch-monitor.service
  netwatch-svc ALL=(ALL) NOPASSWD: /usr/bin/apt-get update
  netwatch-svc ALL=(ALL) NOPASSWD: /usr/bin/apt-get upgrade -y
  netwatch-svc ALL=(ALL) NOPASSWD: /usr/bin/apt-get upgrade --dry-run
[GAP] Sudoers WITHOUT .service suffix -- both forms required. patcher.py calls
  "systemctl restart netwatch-web" (no suffix). Sudoers does exact string matching.
  Without the suffix-less forms, patcher restarts are silently blocked and installs
  appear to succeed but old code keeps running.
  Add to /etc/sudoers.d/netwatch-svc:
    netwatch-svc ALL=(ALL) NOPASSWD: /usr/bin/systemctl restart netwatch-web
    netwatch-svc ALL=(ALL) NOPASSWD: /usr/bin/systemctl restart netwatch-monitor
    netwatch-svc ALL=(ALL) NOPASSWD: /usr/bin/systemctl stop netwatch-web
    netwatch-svc ALL=(ALL) NOPASSWD: /usr/bin/systemctl stop netwatch-monitor
    netwatch-svc ALL=(ALL) NOPASSWD: /usr/bin/systemctl start netwatch-web
    netwatch-svc ALL=(ALL) NOPASSWD: /usr/bin/systemctl start netwatch-monitor
  SESSION: 13 | GOTCHA: SUDOERS EXACT MATCH

1.6 GPG ENCRYPTION (BACKUP)
-----------------------------
[IN INSTALL.SH] GPG public key imported from netwatch-backup-public.asc
  (if present alongside install.sh at install time)
[IN INSTALL.SH] Key trusted via --import-ownertrust
[GAP] netwatch-svc GPG keyring -- required for backup scripts to encrypt.
  The web service (netwatch-svc) runs backup.sh which calls gpg. GPG refuses to
  use a keyring owned by a different user. A dedicated keyring must be created:
    sudo mkdir -p /home/mboynton/netwatch/.gnupg
    sudo chmod 700 /home/mboynton/netwatch/.gnupg
    sudo chown netwatch-svc:netwatch-svc /home/mboynton/netwatch/.gnupg
    gpg --export --armor <recipient@email.com> > /tmp/pubkey.asc
    sudo -u netwatch-svc gpg --homedir /home/mboynton/netwatch/.gnupg \
      --import /tmp/pubkey.asc
    rm /tmp/pubkey.asc
  SESSION: 19 | GOTCHA: GPG KEYRING IS USER-SPECIFIC
  NOTE (Session 26): If gpg reports "unsafe ownership" or "Permission denied" on
  .gnupg, the keyring ownership has drifted. Recovery:
    sudo chown -R netwatch-svc:netwatch-svc /home/mboynton/netwatch/.gnupg
    sudo chmod 700 /home/mboynton/netwatch/.gnupg
  Verify with: sudo ls -la /home/mboynton/netwatch/.gnupg/
  All entries should show netwatch-svc:netwatch-svc ownership.

GPG KEY REPLACEMENT (when generating a new key pair):
  Use when the passphrase is forgotten or the key pair needs to be rotated.
  Old encrypted backups will NOT be recoverable with the new key — that is expected.
  Steps:
  1. On Windows (Kleopatra): delete old key pair, generate new RSA 4096 key for
     mattybusa@gmail.com, export public key as .asc file. Record the passphrase.
  2. SCP public key to Pi (from a separate PowerShell window, not inside SSH):
       scp netwatch-pubkey.asc mboynton@192.168.100.10:/tmp/netwatch-pubkey.asc
  3. On Pi -- replace key in netwatch-svc keyring:
       sudo -u netwatch-svc gpg --homedir /home/mboynton/netwatch/.gnupg \
         --batch --yes --delete-keys mattybusa@gmail.com
       sudo -u netwatch-svc gpg --homedir /home/mboynton/netwatch/.gnupg \
         --import /tmp/netwatch-pubkey.asc
       rm /tmp/netwatch-pubkey.asc
       sudo -u netwatch-svc gpg --homedir /home/mboynton/netwatch/.gnupg --list-keys
  4. Test: sudo -u netwatch-svc bash /home/mboynton/netwatch/backup.sh
  5. Verify decrypt on Windows: scp backup file down, gpg --decrypt, confirm passphrase works.
  SESSION: 27

[GAP] ~/backups/ ownership -- backup scripts run as netwatch-svc and write there.

1.9 GIT VERSION CONTROL
------------------------
Git is initialized in ~/netwatch/ with remote origin pointing to Gitea on ops-pi.
patcher.py automatically commits and pushes on every successful package install.
All git operations run as netwatch-svc.

[MANUAL-ONCE] Initial setup (performed Session 30):

  a) Create Gitea repo: ops-pi.home:3000 → mboynton/netwatch (private, empty)

  b) Create Gitea PAT: Settings → Applications → Generate Token
     Name: netwatch-pi | Permissions: Repository read+write
     NOTE: Copy token immediately — shown only once.

  c) Fix .git directory ownership (git refuses to operate on repos owned by
     a different user):
       sudo chown -R netwatch-svc:netwatch-svc /home/mboynton/netwatch/.git

  d) Configure git identity for netwatch-svc (use full absolute path — tilde
     expands to /nonexistent for netwatch-svc which has no home directory):
       sudo -u netwatch-svc git -C /home/mboynton/netwatch config \
         user.name "NetWatch Pi"
       sudo -u netwatch-svc git -C /home/mboynton/netwatch config \
         user.email "NetWatch.Prime@gmail.com"

  e) Embed PAT in remote URL (passwordless push):
       git -C ~/netwatch remote set-url origin \
         https://mboynton:<PAT>@ops-pi.home:3000/mboynton/netwatch.git

  f) Initial commit and force push (force required because Gitea initialized
     with a README):
       sudo -u netwatch-svc git -C /home/mboynton/netwatch add -A
       sudo -u netwatch-svc git -C /home/mboynton/netwatch commit \
         -m "v3.3.90: Initial commit — full codebase at Session 29 closeout"
       sudo -u netwatch-svc git -C /home/mboynton/netwatch push \
         --force origin main

  g) Update .gitignore to exclude .gnupg/, pkg_update.log, gunicorn.ctl
     then remove already-tracked files from index:
       sudo -u netwatch-svc git -C /home/mboynton/netwatch rm --cached -r .gnupg/
       sudo -u netwatch-svc git -C /home/mboynton/netwatch rm --cached pkg_update.log
       sudo -u netwatch-svc git -C /home/mboynton/netwatch add .gitignore
       sudo -u netwatch-svc git -C /home/mboynton/netwatch commit \
         -m "v3.3.90: Fix .gitignore — exclude .gnupg, pkg_update.log, gunicorn.ctl"
       sudo -u netwatch-svc git -C /home/mboynton/netwatch push origin main

[ONGOING] After v3.3.91 installed: patcher.py handles all commits automatically.
  No manual git steps needed after this. Each package install produces a commit

2.1 PROMTAIL (LOG SHIPPING TO LOKI)
-------------------------------------
Promtail is not part of install.sh. It is a separate infrastructure component.
All steps below are manual. Repeat on any new NetWatch Pi instance.

[MANUAL-ONCE] Download binary (ARM64 -- Pi 3B+ is 64-bit):
  cd /tmp
  wget https://github.com/grafana/loki/releases/download/v3.5.1/promtail-linux-arm64.zip
  unzip promtail-linux-arm64.zip
  sudo mv promtail-linux-arm64 /usr/local/bin/promtail
  sudo chmod +x /usr/local/bin/promtail
  NOTE: Always use linux-arm64, NOT linux-arm (32-bit). See Gotchas.txt.
  SESSION: 17

[MANUAL-ONCE] Create directories:
  sudo mkdir -p /etc/promtail
  sudo mkdir -p /var/lib/promtail

[MANUAL-ONCE] Create service user:
  sudo useradd --system --no-create-home --shell /bin/false promtail
  sudo usermod -a -G systemd-journal promtail
  sudo chown promtail:promtail /var/lib/promtail

[MANUAL-ONCE] Config at /etc/promtail/config.yaml:
  server:
    http_listen_port: 9080
    grpc_listen_port: 0
  positions:
    filename: /var/lib/promtail/positions.yaml
  clients:
    - url: http://192.168.100.230:3100/loki/api/v1/push
  scrape_configs:
    - job_name: journal
      journal:
        max_age: 24h
        labels:
          job: netwatch-pi
          host: netwatch-pi
      relabel_configs:
        - source_labels: [__journal__systemd_unit]
          target_label: unit
        - source_labels: [__journal__hostname]
          target_label: hostname
  NOTE: max_age was 1h initially (Session 17), bumped to 24h (Session 21).

[MANUAL-ONCE] Systemd service at /etc/systemd/system/promtail.service:
  [Unit]
  Description=Promtail log shipping agent
  After=network-online.target
  Wants=network-online.target

  [Service]
  User=promtail
  Group=promtail
  ExecStart=/usr/local/bin/promtail -config.file=/etc/promtail/config.yaml
  Restart=always
  RestartSec=5

  [Install]
  WantedBy=multi-user.target

[MANUAL-ONCE] Enable and start:
  sudo systemctl daemon-reload
  sudo systemctl enable promtail
  sudo systemctl start promtail

[MANUAL-ONCE] Verify:
  sudo journalctl -u promtail -n 10 --no-pager
  Then check Grafana Explore: {host="netwatch-pi"}

SECTION 5: LOG FORMAT REFERENCE
================================================================================
What is sent to Loki from the NetWatch Pi, and in what format:

SOURCE                        FORMAT    NOTES
----------------------------  --------  ----------------------------------------
netwatch-monitor              JSON      structlog, service="monitor" (v3.3.37+)
netwatch-web (app events)     JSON      structlog, service="web" (v3.3.37+)
gunicorn HTTP access log       JSON      StructlogAccessLogger.access() (v3.3.44+)
gunicorn lifecycle messages   PLAIN     Startup/worker messages. access() override
                                        only covers HTTP requests, not internal
                                        gunicorn error logger. Fix pending (S23).
pkg_update.sh                 PLAIN     Shell output via logger/systemd-cat.
                                        Plain text is acceptable -- low volume,
                                        not queried by field.
SSH (sshd)                    PLAIN     System service. Cannot convert without
                                        OTEL Collector middleware. Acceptable --
                                        pattern search with |= works fine.
systemd / kernel / OS         PLAIN     Same as SSH. System services. Acceptable.
Node Exporter                 PLAIN     Startup messages only. No operational value.
Promtail itself               PLAIN     Startup messages only. No operational value.

RULE: Only log sources controlled by NetWatch code are expected to emit JSON.
Third-party system services (SSH, systemd, OS, Node Exporter, Promtail) emit plain
text. This is normal and expected. Loki can search them with |= pattern matching.
The gunicorn lifecycle message gap is the only remaining controllable fix.
