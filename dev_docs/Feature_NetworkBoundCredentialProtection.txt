PROBLEM STATEMENT
-----------------
NetWatch stores sensitive credentials in config.py as plaintext — email
passwords, SMS credentials, SECRET_KEY, and other values written by the
setup wizard. Anyone with filesystem access to the SD card can read these
values directly. This is security audit finding A1.

The threat we are protecting against:
  Physical theft of the Pi or SD card. An attacker reads config.py and
  extracts credentials — potentially including passwords the user has
  reused on other services (email, banking, etc.).

The constraint we must satisfy:
  NetWatch must be able to start and decrypt credentials automatically,
  without any human input, on every boot. Solutions requiring a passphrase
  at boot are not acceptable — if the network is down, NetWatch is idle
  anyway, and we cannot require a person to be present for every restart.

PRIOR ART AND WHY IT DOESN'T FIT
----------------------------------
The concept of tying decryption to network presence is not new. Red Hat's
Tang/Clevis (Network-Bound Disk Encryption, NBDE) does exactly this at the
full-disk level. However Tang requires a dedicated server on the network —
a second Pi, NAS, or similar device. NetWatch cannot depend on this for its
general user base.

TPM (Trusted Platform Module) chips are available as GPIO add-ons for all
Pi models but are not standard hardware and cannot be assumed.

Every other approach examined during the design session was ruled out:
  - Plaintext key on disk: defeats the purpose
  - Key derived from README or source code: source is public on GitHub
  - Router MAC address via ARP: stored in ARP cache on disk; changes with
    hardware replacement
  - Public IP / IP geolocation: dynamic, changes with ISP reassignment,
    city-level accuracy only 55-80%
  - SSID from wpa_supplicant.conf: stored on the SD card (if WiFi)
  - Passphrase at boot: breaks unattended operation
  - Router UPnP UUID: hardware-tied, changes with router replacement

THE NOVEL APPROACH — NETWORK ANCHOR SYSTEM
-------------------------------------------
Instead of a single cryptographic anchor (like Tang's key server), NetWatch
builds a composite key from multiple "anchors" — stable, queryable properties
of the user's home network that are NOT stored on the SD card and are NOT
reachable by an attacker who has stolen the device and taken it elsewhere.

Core insight: the decryption key is never stored anywhere. It is derived
fresh on every boot from live network queries. If the network environment
does not match what was recorded at install time, the key cannot be derived
and secrets remain encrypted.

This is meaningfully different from Tang/Clevis in that:
  - No second server is required
  - No extra hardware is required
  - The system self-configures from whatever anchors are available
  - Protection level scales automatically with the richness of the
    user's network environment
  - Works for a non-technical user with only a modem, router, and phone

ANCHOR CATALOGUE
----------------
Each anchor is a queryable network property. At install time, NetWatch
queries all available anchors and records their values. At boot time, it
re-queries and compares. The composite key is derived from all confirmed
anchors plus an install-time random salt.

ANCHOR 1 — Gateway Reachability (Universal baseline)
  What:     The default gateway IP is reachable via ping/ARP
  Query:    Live ARP/ping at boot — NOT from stored ARP cache
  Stable:   Yes — if the gateway is unreachable, NetWatch has nothing to
            monitor anyway. Degraded mode (no secrets loaded) is correct
            behavior in this case.
  Unique:   No — 192.168.1.1 is the default for millions of routers
  Strength: Baseline — necessary but not sufficient alone
  Notes:    The gateway IP itself is recorded as an anchor value.
            If the Pi boots on a different network (attacker's), the
            gateway IP won't match OR the correct gateway won't respond.

ANCHOR 2 — Subnet / DHCP Range
  What:     The subnet the Pi is on (e.g. 192.168.1.0/24)
  Query:    Read from active network interface at boot
  Stable:   Yes — most users never change their subnet
  Unique:   Low — 192.168.1.0/24 is very common
  Strength: Low on its own, additive with other anchors
  Notes:    Adds meaningful entropy in combination. An attacker would need
            to replicate both the correct gateway IP AND subnet.

ANCHOR 3 — Router Hostname, Default (Moderate)
  What:     The hostname broadcast by the router via mDNS/Bonjour
  Query:    mDNS query at boot (avahi-browse or equivalent)
  Stable:   Yes — default hostnames like "Netgear-A3F2" don't change
            unless the user deliberately changes them
  Unique:   Moderate — model+serial fragment is not globally unique but
            is not shared by millions of devices either
  Strength: Moderate
  Notes:    Most routers broadcast a hostname combining model name with
            a portion of serial number or MAC. This is queryable without
            authentication and is not stored on the Pi's filesystem.

ANCHOR 4 — Router Hostname, Custom (Strong)
  What:     A user-configured router hostname (e.g. "smithfamily-router")
  Query:    Same mDNS query as Anchor 3
  Stable:   Yes — user controls this and is informed to maintain it
  Unique:   High — user-chosen name is effectively unique
  Strength: Strong
  Notes:    Setup wizard recommends users set a custom router hostname
            and explains why. If detected, this anchor contributes
            significantly to key strength. User is told: "If you replace
            your router, set this hostname on the new one."

ANCHOR 5 — SSID (WiFi installs only)
  What:     The SSID of the wireless network the Pi is connected to
  Query:    Live query via iwgetid (NOT from wpa_supplicant.conf)
  Stable:   Yes — users rarely rename their WiFi network
  Unique:   Moderate — "Smith Family WiFi" is fairly unique
  Strength: Moderate
  Notes:    Only available if the Pi is on WiFi. Ethernet-only installs
            skip this anchor. The key point: queried live from the
            wireless interface, not read from the config file on disk.
            An attacker with the SD card who is not on the correct
            wireless network cannot obtain this value.

ANCHOR 6 — UPnP Device UUID
  What:     The UUID broadcast by the router via UPnP/SSDP
  Query:    UPnP SSDP discovery at boot
  Stable:   Yes — UUID is generated at router firmware time, doesn't
            change on reboot. DOES change on router replacement or
            factory reset.
  Unique:   High — UUIDs are designed to be globally unique
  Strength: High
  Notes:    Only available if router has UPnP enabled (most home routers
            do by default). Hardware-tied — user must re-initialize
            anchors after router replacement. Documented clearly.

PROTECTION LEVELS
-----------------
NetWatch reports a protection level based on anchors detected at install.
This is displayed in the setup wizard completion screen and in the Security
settings page.

  MINIMAL   — Gateway IP only
              Better than plaintext. An attacker on a 192.168.1.1 network
              could satisfy this constraint. Not recommended alone.

  LOW       — Gateway IP + Subnet
              Slightly more constrained. Still common network configurations.

  MODERATE  — Gateway + Subnet + Default router hostname
              Meaningful protection. Requires the attacker to replicate
              both the network configuration AND have the same router model
              with the same serial fragment. Realistic protection for most
              home users.

  STRONG    — Any of the above + Custom router hostname OR SSID
              Good protection for most threat models. Requires attacker to
              know and replicate user-specific configuration choices.

  MAXIMUM   — Strong + UPnP UUID
              As strong as this approach can get without TPM or Tang.
              Requires attacker to have the exact same physical router
              hardware in addition to all network configuration details.

KEY DERIVATION
--------------
At install time:
  1. Query all available anchors, record their values
  2. Generate a cryptographically random 32-byte salt (stored on disk)
  3. Concatenate all anchor values in a defined canonical order
  4. Derive encryption key: KDF(anchor_composite + salt)
     Recommended KDF: Argon2id or PBKDF2-HMAC-SHA256 with high iteration
     count (slow derivation is acceptable — only happens at boot)
  5. Encrypt all config.py secrets using AES-256-GCM with derived key
  6. Store: encrypted secrets file + salt + anchor metadata (NOT the key)
  7. Wipe key from memory after encrypting

At boot time:
  1. Re-query all anchors recorded at install time
  2. Compare live values to recorded anchor metadata
  3. If all anchors match: derive key using same KDF, decrypt secrets
     into memory only — never write decrypted values to disk
  4. If any anchor fails to match or is unreachable:
     Start in degraded mode — no secrets loaded, alerting disabled,
     monitoring loop idle. Log the specific anchor that failed.
     This is correct behavior: if the network is wrong, there is
     nothing to monitor and no reason to expose credentials.

SETUP WIZARD INTEGRATION
-------------------------
The setup wizard gains a new step after network configuration:

  "Protecting Your Credentials"

  NetWatch scans your network for security anchors — stable properties
  of your network that are used to protect your credentials. Your
  credentials can only be decrypted when NetWatch is connected to YOUR
  network. If the device is stolen and connected elsewhere, your
  credentials remain encrypted.

  Anchors found:
    ✅ Gateway (192.168.1.1)
    ✅ Subnet (192.168.1.0/24)
    ✅ Router (Netgear-A3F2)
    ❌ Custom router name — not configured
    ❌ WiFi SSID — not applicable (ethernet install)
    ✅ UPnP UUID — found

  Protection level: STRONG

  Recommendation: Set a custom name on your router to reach MAXIMUM
  protection. Here's how: [link to router admin page at gateway IP]

  [Continue with current protection] [I've set a custom name — rescan]

After wizard completion, an email is sent to the admin containing:
  - Protection level achieved
  - List of anchors used (values, not derivation details)
  - Recovery instructions: "If you change your network hardware, log
    into NetWatch and go to Security → Network Anchors → Re-initialize"
  - Recommendation to save this email

RE-INITIALIZATION FLOW
-----------------------
When hardware changes (new router, new subnet, etc.), the user must
re-initialize anchors before making the change, or while they still
have access to the old configuration.

UI location: Security → Network Anchors → Re-initialize

Flow:
  1. User authenticates (already logged in — session required)
  2. NetWatch re-queries the network
  3. Shows new anchor set and new protection level
  4. User confirms
  5. NetWatch decrypts secrets using OLD key (current network must still
     match), then re-encrypts using NEW key derived from new anchors
  6. Sends confirmation email with new anchor summary

Emergency recovery (network has already changed, can't decrypt):
  - User must restore from backup (full backup contains config.py)
  - OR: Admin re-enters credentials manually via setup wizard re-run
  - This is the documented trade-off for network-bound protection
  - Mitigated by: emailing anchor summary at setup time, UI prompts
    to re-initialize BEFORE changing hardware

HONEST LIMITATIONS
------------------
This section must appear in user-facing documentation.

1. NOT foolproof against a sophisticated attacker
   An attacker who steals the SD card AND knows the network configuration
   AND can replicate it (correct gateway IP, subnet, router hostname) could
   potentially satisfy the constraints. The install-time salt prevents
   brute-force attacks against common configurations, but does not make
   this mathematically unbreakable.

2. Hardware changes require re-initialization
   Anchors that are hardware-tied (UPnP UUID) break when hardware is
   replaced. Users are warned at setup and again in the summary email.

3. Does not protect against a running system compromise
   If an attacker gains shell access to the Pi while it is running,
   secrets are decrypted in memory and potentially accessible. This is
   not a problem this feature is designed to solve — that requires
   OS-level hardening (LUKS, etc.).

4. Degraded mode behavior
   If anchors cannot be verified at boot, NetWatch starts without secrets.
   Monitoring is idle. Alerting is disabled. The dashboard shows a clear
   warning. This is intentional and correct — but users should understand
   why it happens.

5. mDNS / UPnP availability
   Some network configurations disable mDNS or UPnP. If these anchors
   cannot be queried, they are simply not used. Protection level degrades
   gracefully rather than failing.

IMPLEMENTATION NOTES (for future session)
------------------------------------------
- New module: netcreds.py — anchor querying, key derivation, encrypt/decrypt
- config.py remains as the human-readable config file; sensitive values
  are replaced with encrypted tokens at setup completion
- A parallel encrypted store (e.g. netwatch.secrets) holds ciphertext
- At runtime, webapp.py and main.py call netcreds.get("EMAIL_PASSWORD")
  rather than reading config.py directly — netcreds handles decryption
  from in-memory decrypted store
- Argon2id preferred over PBKDF2 — better memory-hardness, available
  via argon2-cffi pip package
- AES-256-GCM for encryption — authenticated encryption prevents
  tampering with the ciphertext
- All anchor queries must have timeouts — boot should not hang if mDNS
  or UPnP is slow to respond
- Anchor query library candidates:
    mDNS:  python-zeroconf (pure Python, no system deps)
    UPnP:  async-upnp-client or raw SSDP socket query
    WiFi:  subprocess call to iwgetid
    ARP:   subprocess call to arping or read /proc/net/arp
- This feature touches: install.sh, setup wizard, webapp.py, main.py,
  config_schema.py, and requires new netcreds.py module
- Requires design brief review before implementation begins
