SECURITY AUDIT — DISCUSSION ITEMS (no code)

M4 — Config export credential risk: Reviewed and found the export is already
  client-side encrypted (.nwcfg format, passphrase required). Risk was lower than
  the original audit suggested. Decision: add a warning note to the export modal
  and close. Shipped in v3.4.44.

M5 — Per-IP rate limiting: Discussed Flask-Limiter (Option A). Decided to accept
  and defer. Single-user install means per-user lockout already closes the attack —
  spreading across IPs gains the attacker nothing. Revisit in mBase with
  Flask-Security-Too.

C4 — Package signing: Extended design discussion covering GPG, Minisign, Python-
  native Ed25519, and ops-pi-as-signing-authority. Ultimately concluded that SHA-256
  verification on GitHub-hosted releases (already implemented) is what most open
  source projects rely on and is sufficient for this distribution model. Closed.
  No code needed.

SECURITY HARDENING — v3.4.44

  C2 — DB restore validation: admin_restore_db() now checks all 16 expected tables
    are present in the uploaded file, plus column signatures on users, roles, and
    system_settings before overwriting the live database. Rejects corrupt or wrong-
    database uploads with a clear error message. webapp.py.

  C3 — Destructive SQL guard: patcher.py now blocks DROP, DELETE, and TRUNCATE at
    both preview time and apply time unless ALLOW_DESTRUCTIVE_SQL=True in config.
    Preview shows a clear rejection with instructions. Apply has a belt-and-suspenders
    hard guard. Auto-resets to False after any destructive install completes.
    New config key ALLOW_DESTRUCTIVE_SQL (bool, default False, Security section)
    added to config_schema.py. Both services restart.

  H3b — Reset token replay block: request_password_reset() in auth.py now refuses
    to issue a new token if a valid one already exists. Returns "reset_pending"
    (treated as success by caller — no signal leaks to requester). Prevents inbox
    flooding and rolling fresh-token attacks.

  M4 — Config export warning: export passphrase modal text updated to note the file
    contains credentials. config_editor.html.

HARDWARE BRING-UP (see Feature_HardwareBringup_Session40.txt for full detail)

v3.4.45 — button.py rewritten to use lgpio: RPi.GPIO edge detection broken on
  new Pi OS gpiochip kernel driver. lgpio uses /dev/gpiochip, works correctly
  as netwatch-svc. Callback signature changed to (chip, gpio, level, timestamp).
  lgpio installed in venv manually (swig + liblgpio-dev required).

v3.4.46 — Button long press changed from modem-only to router-only reset.
  Rationale: modem-only rarely makes sense (router downstream), router-only is
  the more useful deliberate action.

v3.4.47 — Button long press + SystemSetup.txt update: docs updated with gpio group
  membership requirement, lgpio venv gap (GAP-10), and ops-pi.home hosts entry.
  Supersedes v3.4.46 — install only v3.4.47.

INFRASTRUCTURE FIXES (manual, not packaged)

- netwatch-svc added to gpio group: RPi.GPIO output requires /dev/mem access.
  Without gpio group membership, relay GPIO calls silently fail — monitor logs
  show reset cycle completing but relay never fires. Fix: usermod -a -G gpio
  netwatch-svc. Already in install.sh for fresh installs.

- lgpio installed in venv: button.py uses lgpio but it was not in the venv on
  older installs. Required swig + liblgpio-dev (apt) then pip install lgpio.
  Documented as GAP-10 in SystemSetup.txt.

- ops-pi.home DNS: git remote uses ops-pi.home but mDNS resolves ops-pi.local.
  Gitea TLS cert issued for ops-pi.home — cannot use .local hostname. Fixed with
  static /etc/hosts entry: 192.168.100.230 ops-pi.home. Gitea push confirmed
  working after fix.


WHAT WAS ATTEMPTED BUT FAILED
-------------------------------
- Initial C4 design went deep into PKI (Ed25519, ops-pi signing server, key rotation)
  before stepping back to ask what other open source projects do. The answer (SHA-256
  on GitHub releases) made the entire design unnecessary. Should have asked that
  question first.

- lgpio not in venv: first pip install attempt failed with "swig not found", then
  after installing swig failed with "cannot find -llgpio". Required two separate apt
  installs (swig, liblgpio-dev) before pip install succeeded. Then failed with
  Permission denied — required sudo pip install.

- Relay not firing from web UI: initially suspected simulation mode, permissions,
  config mismatch, and path issues before identifying the root cause (gpio group
  membership). The monitor logs showed reset cycles completing with correct log
  messages because the log statements are before/after the GPIO calls — not inside
  them. The GPIO calls executed silently without effect.

- Button wiring appeared incorrect (physical pin numbering vs BCM numbering concern)
  but turned out to be correct — pin 14 is GND, pin 15 is BCM GPIO 22.

- Relay DIP switch was set incorrectly on the new hardware (user caught this and
  corrected it — not a code issue).


DECISIONS MADE
---------------
- M5 accepted/deferred: per-user lockout is sufficient for single-user installs.
  Per-IP limiting adds no meaningful protection. Revisit in mBase.
- C4 closed: SHA-256 on GitHub releases is the right level for this project.
  No key management infrastructure warranted at this stage.
- C3 implemented as guardrail not security control: ALLOW_DESTRUCTIVE_SQL prevents
  accidental data loss, not malicious attacks. Consistent with overall threat model.
- Button long press → router reset (not modem). Modem-only reset rarely makes sense
  since router is downstream. Router-only is the more deliberate useful action.
- v3.4.46 superseded by v3.4.47 within same session — install only .47.
- ops-pi.home in /etc/hosts rather than changing git remote — TLS cert is issued
  for .home, cannot use .local without cert mismatch.
- End-of-session context doc + CurrentTask.txt update in closeout package.
