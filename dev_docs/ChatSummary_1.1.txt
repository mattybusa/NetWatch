WHAT WAS ACCOMPLISHED
---------------------
- Architecture decision: Flask + SQLite chosen over InfluxDB + Grafana. User was open to
  recommendation. Flask/SQLite selected for Pi 3B+ RAM constraints (Grafana alone ~300-400MB);
  Chart.js added for visualization. This is the foundational decision that all subsequent
  development builds on.
  (see Feature_Architecture_Decisions.txt)

- Initial code generation (pre-v2.0): Full modular NetWatch codebase generated across
  multiple context compaction events. The session went through two rebuild cycles before a
  clean 16-file modular structure was established. Files: monitor.py, network.py, relay.py,
  button.py, database.py, alerts.py, main.py, webapp.py, auth.py (later), config.py,
  templates/ (index.html, login.html, etc.), static/, systemd service files.
  (see Feature_InitialCodebaseAndDeployment.txt)

- OS/Pi setup: User started with old SD card image, formatted and provisioned from scratch.
  Hardware: Pi 3B+, USB SSD adapter (SSD recommended over SD for SQLite write durability).
  Network: 192.168.17.0/24 primary subnet, 192.168.100.0/24 wired automation, 
  192.168.110.0/24 wireless automation. Pi static IP assigned at 192.168.100.10 (wired)
  and 192.168.110.10 (wireless). Username: mboynton. Home dir: /home/mboynton/netwatch/.
  
- Deployment: Files transferred via SCP from Windows to Pi. Initial service startup.
  Button GPIO edge detection error encountered and resolved (service file fix).
  Services: netwatch-monitor and netwatch-web (systemd units).
  (see Feature_InitialCodebaseAndDeployment.txt)

- Authentication system (v2.0): Role-based auth with session management. Built-in roles:
  Admin (full access, 8hr timeout), Monitor (read-only, never times out), Operator
  (dashboard+controls, 24hr timeout). Roles are user-configurable: name, permissions, and
  session timeout all editable from admin. Users assigned a role. Passwords hashed. Login
  page, forced password change on first login, admin user management UI.
  (see Feature_AuthAndRoles.txt)

- Themes and layouts (v2.0): 4 themes (Dark Blue, Dark Green/terminal, Light/day,
  High contrast), 4 layouts (Dashboard-first, Wide sidebar, Comfortable, Compact). Theme
  and layout stored per-user in the DB. Applied via CSS classes on <body> driven by Jinja2.
  Single base.html template extended by all pages; themes.css holds all variables.
  (see Feature_ThemesAndLayouts.txt)

- User preferences architecture: Theme + layout columns on users table. Webapp reads
  at login and stores in session. Appearance popover (top-right or bottom-left of sidebar
  depending on nav style) allows live switching without full page reload. Preferences
  persist across sessions via DB.
  (see Feature_ThemesAndLayouts.txt)

DECISIONS MADE
--------------
- Flask + SQLite over InfluxDB + Grafana: permanent architectural decision for all future
  development. SQLite for time-series metrics with Chart.js for visualization.

- Modular file structure: 16 separate files over a single consolidated module. Allows
  targeted patching and cleaner service restart logic.

- SSD over SD card for production: SQLite write frequency (logging every ~30s) would
  wear out SD cards. Pi boots from SD, app runs from USB SSD. User noted desire to migrate
  eventually; production unit running on SSD from this point forward.

- Static IP: Pi at 192.168.100.10 (wired) and 192.168.110.10 (wireless). Both subnets
  give dashboard access.

- Role system architecture: Roles define permissions + session timeout. Users inherit from
  role. This allows a Monitor display role with infinite timeout and read-only access — the
  user's explicit use case.

- Relay wiring: NC (normally closed) was determined to be the appropriate wiring approach
  for power-cycle use (modem and router). Active-HIGH relay logic confirmed.
  [NOTE: Relay logic debugging with active-HIGH/NC wiring occurred later in 1.2 — see
  Feature_RelayAndControls.txt]

- Theme/layout approach: CSS class on <body> tag, not separate templates. One themes.css.
  Adding a new theme = adding CSS variables only.

- File uploader and config editor: designed and planned in this sub-session. User requested
  "intelligent zip-based patch system" for future updates. Implementation deferred to 1.2.

- Context doc: No formal AI context document established yet in this sub-session.
  That system was created later.
