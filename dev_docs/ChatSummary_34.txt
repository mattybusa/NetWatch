v3.3.98 — Check for Updates button on /admin/patch (see Feature_ReleaseAndDist.txt).

  Visible only when UPDATE_CHECK_URL is configured.

v3.3.99 — Fix check_now on production Pi (see Feature_ReleaseAndDist.txt).
  requests library was imported locally inside the function with try/except ImportError.
  On production Pi, requests was never installed (Pi predates install.sh). The module-
  level import pattern from main.py (_requests = None fallback) was adopted. Root cause
  confirmed: production Pi was hand-built before install.sh existed; requests was in
  install.sh but never actually installed. Test Pi (fresh install.sh) had it correctly.
  Manual fix on production: sudo -u netwatch-svc ~/netwatch/venv/bin/pip install requests.
  GAP-9 added to SystemSetup.txt.

  constant passed from route to template; header, cells, colspan, and pollGitStatus()

  (3) Gotchas.txt updated with BUILD SCRIPTS INVOKED BY WEB SERVICE entry.

v3.4.5 — Fix build_release.sh output path: $HOME resolves to /nonexistent for

  /home/mboynton/backups/releases. Also fixed release_download route in webapp.py

v3.4.9 — Fix publish_release asset upload: existing asset with same filename causes

WHAT WAS ATTEMPTED BUT FAILED
------------------------------
- v3.3.98 check_now returned "requests library not available" on production. Root
  cause diagnosed incorrectly as an import failure — actual cause was requests
  genuinely not installed in production venv (hand-built Pi, predates install.sh).
  Module-level import fixed the pattern but not the underlying missing package;
  manual pip install was the actual fix.

- First attempt to save config.py with nano failed with "Permission denied" — file
  owned by netwatch-svc. Used sudo tee -a instead.

- Build Release button did nothing (v3.4.3): stray closing brace in admin.html JS
  killed script block before doBuildRelease was defined. Fixed in v3.4.4.

- build_release.sh wrote output to /nonexistent/backups/releases (v3.4.3): $HOME
  for netwatch-svc. Fixed in v3.4.5 with hardcoded absolute path.

- GitHub Publish returned 401 Bad Credentials (v3.4.3): remote URL format was
  mattybusa:TOKEN@github.com — regex captured mattybusa:TOKEN as the entire PAT
  instead of just TOKEN. Fixed regex in v3.4.6 to handle both formats.

- releases/latest.json permission denied (v3.4.3): file was created manually by
  mboynton in Session 32, owned by mboynton. netwatch-svc couldn't write it.
  Manual chown fix applied; v3.4.7 adds chown after write for future runs.

- GitHub Release 422 already_exists on Publish retry (v3.4.7): fixed to fetch
  existing release instead of failing.

- manifest.json was empty/invalid JSON on test Pi install (v3.4.8): bash heredoc
  with quoted delimiter (<<'PYEOF') does not expand bash variables. MANIFEST_ACTIONS
  was never interpolated. Fixed by writing to temp file and passing as Python argv.

- GitHub asset upload 422 already_exists (v3.4.9): fixed by deleting existing asset
  before uploading replacement.

- Distribution/installer design discussion went too broad before testable foundation
  existed. Pulled back to build_release.sh and end-to-end test first. Correct approach.


DECISIONS MADE
--------------
- requests missing from production venv documented as GAP-9 in SystemSetup.txt.
  Rule: any new server-side HTTP call should verify requests is in the production venv,
  not just in install.sh.

- DEVELOPMENT_SYSTEM config key: ui_hidden=True, set manually in config.py, never
  shown in Config Editor. Release Manager only visible when True. Regular users never
  see it.

- "Share NetWatch" card removed entirely. The old generate_distrib and generate_update_pkg
  routes were incompatible with the patcher-format release zip model and removed.

- GitHub PAT scope: existing "repo" scope covers releases write. No PAT changes needed.

- releases/latest.json ownership: file must be owned by netwatch-svc. Manual chown
  applied in Session 34; v3.4.7 adds automatic chown after every write.

- MIGRATIONS.md: cleared at v1.0.0 reset point since all 3.x.x entries are pre-public
  and already baked into init_db() for fresh installs. Public users only ever upgrade
  from v1.0.0+, so pre-1.0.0 migration entries are irrelevant and would cause incorrect
  SQL application if left in.

- Version reset to v1.0.0: deferred. Not ready yet — need more testing, install path
  work, and documentation before declaring public release quality.

- Test Pi left at v3.4.6 (public release version) deliberately. Future releases tested
  against it via update checker. Do not install further development packages on test Pi.

- Production only gets development packages going forward. Test Pi only gets public
  releases via update checker.

- Distributable installer approach: Option A (Imager advanced options + one curl command)
  for first release. Cloud-init (Raspberry Pi OS Trixie, Nov 2025) is the modern path —
  user copies user-data file to boot partition, Pi installs itself on first boot. No
  Foundation involvement required. Design deferred to future session.

- build_release.sh uses Python (passed via temp file) for manifest generation, not bash
  heredocs. Shell heredocs with quoted delimiters do not expand variables.

- End-of-session context doc at v3.4.9 reflects all session work.
