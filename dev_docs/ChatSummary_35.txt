DESIGN DISCUSSION — Self-healing package dependency system
  Extended discussion about making install.sh resilient to future package renames.
  Considered: try/fallback, distro detection, pip-only, CI testing, OS pinning.
  Investigated: Debian Package Tracker API (no machine-readable successor endpoint),
  apt-cache Replaces: field (not reliably populated), Madison API
  (api.ftp-master.debian.org/madison — confirmed working JSON API).
  
  Key insight: source package name (e.g. "libgpiod") is stable across renames.
  Binary name (libgpiod2 → libgpiod3) changes. Query API at install time using
  source name, filter to "stable" suite, match binary pattern. Authoritative,
  no fuzzy matching, no external dependency beyond curl.
  
  Final design: dynamic resolution in install.sh only. No database table, no
  canary, no release builder integration. Loud failure if API unreachable.
  Implemented in v3.4.11.

DESIGN DISCUSSION — GitHub sync strategy
  Discovered publish_release only committed latest.json and pushed that one commit
  to GitHub. Patcher auto-commits to Gitea after every install but never to GitHub.
  Result: GitHub main branch was a patchwork of files from different points in time.
  Fix: push all accumulated commits unconditionally on publish. Git handles the diff
  — sends everything GitHub hasn't seen since last push. Fixed v3.4.12.


WHAT WAS ATTEMPTED BUT FAILED
------------------------------
- Dynamic package resolution (v3.4.11 first attempt): log() calls inside
  resolve_debian_package() went to stdout, polluting LIBGPIOD_PKG variable with
  log timestamp + package name. Fixed by redirecting log calls to stderr.
- GitHub sync (v3.4.12 first attempt): tried git add -A + commit before push.
  If nothing new to commit (patcher already committed everything), git exits 1,
  check=True raises exception, push never happened. Fixed by checking git status
  first and only committing if staged changes exist, then pushing unconditionally.
- Debian Package Tracker REST API: returns 404. Not a machine-readable API.
  sources.debian.org API exists but is for source browsing, not binary successor
  lookups. Madison API (confirmed working) was the right tool.


DECISIONS MADE
--------------
- Dynamic package resolution: query api.ftp-master.debian.org/madison at install
  time using source package name. No hardcoded binary names for unstable packages.
  Only libgpiod currently flagged as unstable — python3-lgpio and all others are
  stable Debian names.
- GitHub main branch strategy: publish_release pushes all accumulated patcher
  commits to GitHub. GitHub stays current only at publish time, not after every
  install. This is correct and expected behavior.
- install.sh curl approach: self-contained — downloads its own codebase from
  the GitHub release zip. No manual zip extraction required by user.
- Simple network WiFi monitoring: WIFI_GATEWAY set to LAN_GATEWAY in config,
  checked via WIFI_INTERFACE independently from LAN check. WiFi check only
  activated if wifi_interface field is populated in wizard.
- Canary / database tracking for package names: decided against. Installer is
  fully dynamic — no tracking needed. Loud failure is sufficient safety net.
