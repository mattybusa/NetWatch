BUG: ADMIN PASSWORD RESET — FORCE-CHANGE NOT TRIGGERING + MISSING CHECKBOX
===========================================================================
Type: Bug (two related issues)
Date: 2026-03-14
Session: 14
Final outcome: Complete — both issues fixed, confirmed working by user.


BACKGROUND
----------
Admin users can reset any user's password via Users & Roles → Reset Password modal.
The intended behavior: admin sets a temporary password, checks a box to require the
user to change it on next login (must_change_pass=1), and the user is redirected to
the change-password page immediately after logging in with the temp password.

Two problems were reported in Session 14:
1. The "require password change" checkbox was gone from the dialog — it had been
   replaced with static explanatory text, removing the option to skip the forced
   change (needed for testing with known passwords).
2. Even when the dialog said the user would be forced to change their password, they
   were not redirected on login — the forced-change flow never triggered.


WHAT WAS TRIED AND FAILED
--------------------------
No failed approaches this session. Root cause was identified by tracing the code
path before writing any fix.


ROOT CAUSE ANALYSIS
-------------------
Issue 1 (missing checkbox):
admin_users.html Reset Password modal had hardcoded static text:
  "They will be required to change it on next login."
and the JS sent { force_change: true } unconditionally. The checkbox had been
removed at some point in prior session history with no record of why.

Issue 2 (must_change_pass not triggering):
The code path on admin reset was:
  1. auth.change_password(user_id, new_password)
     → sets password_hash=hash, must_change_pass=0  ← clears the flag
  2. auth.update_user(user_id, must_change_pass=True)
     → sets must_change_pass=1  ← sets it back

So the DB ended up at must_change_pass=1 correctly after admin reset.

But then when the user logged in with that password:
  authenticate() real-password login path (auth.py line ~299):
    UPDATE users SET last_login=?, ..., must_change_pass=0 WHERE id=?
  → cleared the flag unconditionally on every real-password login

The flag was wiped before the login handler (webapp.py) ever read it to decide
whether to redirect to change_password. The user was then logged in normally with
must_change_pass=0 in the DB, so no forced-change redirect occurred.

The reason this is subtle: the flag was correctly written to 1 in the DB between
the admin action and the login. It only got wiped at the moment of login itself.
Checking the DB between steps would have shown it set correctly.


WHAT WORKED
-----------
Fix 1 — Restore force-change checkbox (admin_users.html):
- Replaced static text with a checkbox input (id="rp-force-change", checked by default)
- Label: "Require password change on next login"
- openResetPass() now resets the checkbox to checked every time the modal opens,
  so it can't accidentally persist an unchecked state from a previous open
- resetPassword() reads the checkbox value and passes force_change as a boolean
  in the API request payload

Fix 2a — New auth.admin_reset_password() (auth.py):
  def admin_reset_password(user_id, new_password, force_change=True):
- Sets password_hash and must_change_pass in a single atomic UPDATE
- Does not go through change_password() — that function has the contract of clearing
  must_change_pass (correct for user-initiated changes, wrong for admin resets)
- force_change=True is the default; passing False sets must_change_pass=0 (permanent
  password with no forced change)
- Replaces the old two-step change_password() + update_user(must_change_pass=True)

Fix 2b — Remove must_change_pass=0 from authenticate() real-password login path (auth.py):
- The real-password login UPDATE no longer includes must_change_pass=0
- The flag is now only cleared by change_password() when the user completes their
  own change (correct invariant: DB value is authoritative; authenticate() should
  not silently reset state)
- The used_reset_token path was already correct (it never cleared must_change_pass)

Fix 2c — webapp.py api_reset_user_password() updated:
- Now calls auth.admin_reset_password(user_id, new_password, force_change=force_change)
- Reads force_change from request payload (defaults True if not supplied)
- Removed the old two-step change_password() + update_user() sequence

Shipped in: v3.3.35

------------------------------------
None. Both issues fully resolved.
