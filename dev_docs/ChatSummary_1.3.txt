ALERTS_ENABLED AS SYSTEM-WIDE MASTER SWITCH (clarification, v3.2.36):
The relationship between config.ALERTS_ENABLED and per-user alert subscriptions was
clarified in a detailed architecture discussion (v3.2.31-v3.2.37 transcript):

ALERTS_ENABLED in config.py = global infrastructure kill switch ("can the system
send anything at all?"). It holds the SMTP credentials. Turning it off disables all
outbound alert delivery regardless of any user's subscription preferences.

Per-user subscriptions = "what does this user want to receive?" (dimmer switches).
These are independent of the global toggle.

A soft "alerts disabled" banner is shown to account holders in My Account → Alert
Subscriptions when ALERTS_ENABLED is False, so users understand why they're not
receiving alerts even if they have subscriptions enabled.

Test button behavior:
- When email unverified: Test button hidden (cannot send to unverified address)
- When ALERTS_ENABLED = False: clicking Test returns "alerts are disabled" message
These are separate conditions with separate behaviors. Fixed in v3.2.38.

VERIFY BUTTON ALWAYS VISIBLE WHEN UNVERIFIED (v3.2.38):
The email/phone verification button was being gated on "resave preferences" — it
disappeared and only reappeared after saving the preferences form. Fixed: verify button
always visible when email/phone is not verified, regardless of save state.


ADDITIONAL FINDINGS — Bug_ConfigEditorSave.txt
-----------------------------------------------

ALERTS_ENABLED TOGGLE ARCHITECTURE CLARIFICATION:
The config editor boolean toggle uses a hidden <input type="checkbox"> and a visual
<span> track. The double-fire bug (v3.2.38-v3.2.41) was caused by:
1. onclick on the <span> (visual track) fires toggleBool(), which sets cb.checked = !cb.checked
2. The <label> wrapping the hidden checkbox also receives the click event and flips
   cb.checked again — restoring the original value

Net effect: every toggle click resulted in a no-op (value flipped twice back to original).
Fix: stopPropagation() on the span click handler to prevent the label from receiving
the event. This is a standard HTML form interaction bug with custom-styled checkboxes.

DIAGNOSTIC ROUTE CLEANUP (v3.2.42):
/api/config/diag/alerts debug route was added to expose in_memory vs on_disk config
values during the config editor debugging arc. After the fix was confirmed working,
this route and all associated diagnostic print statements were cleaned up from webapp.py
in v3.2.42. Rule: diagnostic routes are always temporary — clean up before moving on.

ADDITIONAL FINDINGS — Bug_MetricsChartAndAdminCollapse.txt
----------------------------------------------------------

DOMContentLoaded TIMING — initCharts() ORDERING (v3.2.50 arc):
During the diagnostic arc for the blank 1H chart, a second theory was explored:
The Pi's metrics.html was calling initCharts() and loadData() directly at script
execution time (bottom of <body>). The canonical version wrapped these in
DOMContentLoaded. The difference: direct execution at </body> runs after the DOM is
parsed but before all deferred subresources. DOMContentLoaded fires after all deferred
scripts and subresources, including CSS. initCharts() calls cssVar() which reads CSS
custom property values — if CSS hasn't loaded, cssVar() returns empty string and the
chart renders with default colors or zero dimensions.

This was a real drift between Pi and canonical. The Pi had accumulated a version of
metrics.html from an earlier session that lacked the DOMContentLoaded wrapper.
However, fixing this drift was NOT the cause of the blank chart — the monitor crash
was the real cause. The DOMContentLoaded fix was included anyway as a correctness fix.

CLOCK DRIFT THEORY — WHY IT WAS PLAUSIBLE:
The v3.2.50 arc's wrong clock-drift diagnosis was understandable given the symptoms.
At the point the theory was formed, the only visible evidence was:
1. 6H chart showed data correctly
2. 1H chart showed y-axis of 0-1 with no plotted points
3. The monitor service appeared to be running (journalctl showed no obvious crash loop)
The crash loop was subtle: init_db() crashed on startup, but systemd restarted the
monitor successfully after each crash so journalctl showed normal restart behavior with
very short "active" windows. The monitor was starting, crashing in init_db() before
writing any health records, then being restarted. This looked identical to a running
monitor with no records matching the query window — hence the clock drift hypothesis.

The decisive evidence that disproved clock drift: the user's pushback — "this has worked
until just recently, it was caused by a change we did." This directed attention to what
changed in v3.2.49, which led directly to the init_db() multi-statement c.execute() bug.

STYLED HTML THEME SELECTION (v3.2.23-v3.2.24, additional detail):
The changelog HTML theme color is written in at generation time from the installing
user's current theme. If the user is on dark-blue theme when installing, the changelog
header uses dark-blue accent colors. This was an intentional design choice discussed
before building: "Themed header + white body for printing" was the user-selected style.
The print-safe reasoning: the themed header makes it visually identifiable on screen
as a NetWatch artifact. The white body ensures ink is not wasted if anyone prints it.
The style was chosen over three alternatives (A, B, C) that were previewed as PDF
samples during the v3.2.5-v3.2.7 backlog planning session. Style A with white body
was selected.

CHANGELOG VERSION STRING NORMALIZATION (v3.2.25 era):
The packaging library's version comparison treated "3.2.24b" as a pre-release of
"3.2.24" (older). This was discovered when the patcher flagged a 3.2.24b install as
out-of-order. From this point forward: no letter suffixes in version numbers.
Use dotted numerics only: 3.2.24.1 or 3.2.25, never 3.2.24a or 3.2.24b.
This was also applied retroactively: the patcher's version comparison logic was
updated to handle dotted numerics only, rejecting letter suffixes with a clear error.
