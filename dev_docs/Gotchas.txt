2026-03-02 | v3.2.50f

Python's sqlite3 module raises ProgrammingError if a single c.execute() call contains

2026-03-11 | v3.3.2–v3.3.4
PURE ASCII ONLY IN ALL .sh FILES
Shell scripts containing Unicode box-drawing or symbol characters (e.g. = - + arrows)
cause bash syntax errors on the Pi when transferred via SCP or Dev Docs upload,
even if the file is valid UTF-8. The Pi's bash parser chokes on multi-byte characters
in certain contexts. Rule: ALL .sh files must use pure ASCII only. No decorative
box-drawing, no Unicode arrows, no Unicode checkmarks. Use +, -, =, and plain text
equivalents instead.

2026-03-13 | v3.3.19
JINJA2 NESTED BLOCKS CAUSE DOUBLE RENDERING
If {% block scripts %} is opened inside {% block content %} in a child template, but
base.html defines them as separate sequential blocks, Jinja2 renders the scripts content
TWICE -- once inline within content, and again at the standalone {% block scripts %}

a top-level sibling, not nested inside another block, unless base.html also nests them
the same way.

2026-03-13 | v3.3.19-v3.3.21
CSS CUSTOM PROPERTIES: body BEATS :root FOR INHERITANCE
CSS custom properties (variables) resolve by DOM inheritance, not just specificity. If
body.theme-* sets --bg-card on body, that value wins over :root { --bg-card } because body
is a closer ancestor to all elements than :root. Setting variables on :root does NOT
override variables set on body. To override body.theme-* variables: use a selector with
equal or higher specificity (e.g. body[class]) placed AFTER themes.css in the <head> so
source order breaks the tie.

2026-03-13 | v3.3.25
PATCHER LEAVES TEMP .pyc FILES IN /tmp/__pycache__/ -- NEVER CLEANED UP
FIXED in v3.3.53 -- this entry retained for historical reference.
patcher.py performs a syntax check by compiling each Python file to a temp .pyc in
/tmp/__pycache__/. It never cleans up these files. They accumulate across every package
install and are owned by whatever user ran the install. If the directory exists with
wrong ownership/permissions when a new user (e.g. netwatch-svc via web service) tries
to write there, the patcher reports a misleading "syntax error in filename.py" when the
real error is a permission failure on the temp file. Workaround: sudo chmod 777
/tmp/__pycache__. Fixed in v3.3.53 by using tempfile.mkstemp() with an explicit cfile
argument so py_compile never touches /tmp/__pycache__/, and cleaning up both temp files
in a finally block. PermissionError now reported accurately instead of as a false syntax
error. If /tmp/__pycache__ accumulated files before v3.3.53: sudo rm -rf /tmp/__pycache__

2026-03-13 | v3.3.25
SERVICE ACCOUNT WITH --no-create-home: expanduser() BREAKS -- USE __file__ PATHS
When a Python script runs as a system account created with --home-dir /nonexistent
--no-create-home, os.path.expanduser("~") resolves to /nonexistent. Any path built
with expanduser() will fail when the code tries to create or access that path.
All Python files that run as the service account (main.py, theme_manager.py) must use
os.path.dirname(os.path.abspath(__file__)) to build paths relative to the script
location instead. Shell scripts that run as mboynton over SSH are unaffected -- $HOME
resolves correctly for the interactive user.

GOTCHA: PACKAGE ZIP STRUCTURE MUST MIRROR TARGET PATH STRUCTURE
Session 10 | 2026-03-13

Files in a package zip must be placed inside a subdirectory matching their target
path prefix. For example, dev_docs files must be inside a dev_docs/ folder in the
zip, not at the zip root. The patcher validation step checks zip structure and will
hang on "validating" indefinitely if files are at the wrong level.

Manifest.json always sits at the zip root. Everything else mirrors its target path.

Correct structure:
  manifest.json          (root)
  dev_docs/
    ChatSummary_10.txt
    Gotchas.txt
    ...
  templates/
    some_template.html
  static/
    some_file.js

Wrong structure (causes hang on "validating"):
  manifest.json
  ChatSummary_10.txt     (should be inside dev_docs/)
  Gotchas.txt
  ...

The _find_in_zip function matches by basename, so the patcher CAN find files
at the wrong level -- but an earlier validation step rejects the package before
_find_in_zip is ever reached. Always match zip structure to target path structure.

GIT ON A SERVICE-ACCOUNT-OWNED DIRECTORY
-----------------------------------------
~/netwatch is owned by netwatch-svc:netwatch-svc with permissions drwxr-x---.
Running git init as mboynton fails with "Permission denied" even after adding
mboynton to the netwatch-svc group, because the directory itself lacks group
write permission.

Fix requires two steps:
1. sudo usermod -a -G netwatch-svc mboynton (then log out and back in)
2. sudo chmod g+w ~/netwatch

Also required:
  git config --global --add safe.directory /home/mboynton/netwatch
  (git refuses to operate on directories owned by a different user without this)

GIT FIRST PUSH WITH INITIALIZED REMOTE
----------------------------------------
When you create a repo on Gitea with "Initialize repository" checked (adds
README and LICENSE), and then try to push a local repo for the first time,
git rejects the push because the remote has commits the local doesn't have.

Fix:
  git config --global pull.rebase false
  git pull origin main --allow-unrelated-histories
  (resolve any conflicts, commit the merge)
  git push -u origin main

GITEA INITIAL SETUP WIZARD REWRITES app.ini
---------------------------------------------
The Gitea setup wizard rewrites app.ini without section headers, breaking
the config. After running the wizard:
1. Verify /etc/gitea/app.ini still has [server], [database], etc. headers
2. If headers are missing, rewrite the file manually with correct structure
3. Set INSTALL_LOCK = true in [security] to prevent wizard from reappearing
4. Permissions must be temporarily relaxed for the wizard to write the file:
   sudo chmod 777 /etc/gitea && sudo chmod 666 /etc/gitea/app.ini
   Lock back down after install: sudo chmod 750 /etc/gitea && sudo chmod 640 /etc/gitea/app.ini

NETPLAN CONNECTION NAME ON RASPBERRY PI OS LITE 64-BIT
--------------------------------------------------------
Newer Raspberry Pi OS Lite 64-bit uses NetworkManager with netplan, not dhcpcd.
dhcpcd.service does not exist. nmcli is the correct tool.
Connection name is NOT "Wired connection 1" -- check with: sudo nmcli con show
Actual name on a fresh Pi is typically "netplan-eth0".
Static IP command:
  sudo nmcli con mod "netplan-eth0" ipv4.addresses X.X.X.X/24 \
    ipv4.gateway X.X.X.X ipv4.dns 8.8.8.8 ipv4.method manual
  sudo nmcli con up "netplan-eth0"

GITEA BUILT-IN SSH VS SYSTEM SSH PORT CONFLICT
------------------------------------------------
Gitea's built-in SSH server and the system SSH daemon both want port 22.
Setting SSH_PORT = 2222 in app.ini does not automatically make Gitea's
built-in SSH server bind to 2222 -- it may simply not start.
Workaround: use HTTPS with stored credentials (git credential.helper store).
This is fully functional and passwordless after first push.
Gitea SSH passthrough (using system SSH to route git connections to Gitea)
is an alternative but requires additional configuration.

OPENSSL SERIAL FILE PERMISSIONS
---------------------------------
When signing a certificate with openssl x509, if the CA serial file (.srl)
is not writable by the current user, the signing produces a 0-byte output
file with no clear error message -- just a permissions warning in stderr.
Fix: sudo chmod 664 ~/netwatch/certs/ca/netwatch-ca.srl before signing.

HTTP FILTER BLOCKS "session" AS A HEADER KEYWORD IN FETCHED FILES
When Claude fetches a file over HTTP (e.g. from Gitea via Claude in Chrome),
any line containing certain keywords like "session" can be intercepted and
blocked by a JS security filter in the browser extension, treating it as an

check for HTTP header keywords appearing on their own line: "session",
"cookie", "authorization", "content-type", etc. Misspell the keyword in the
doc and add a NOTE comment explaining why. Never misspell keywords in actual
code files -- only in documentation.

CLAUDE IN CHROME OPERATES WITH THE USER'S FULL BROWSER SESSION PRIVILEGES
Claude in Chrome is not a privileged agent with its own credentials -- it acts as
the logged-in user, in their browser, with their active sessions. This means Claude
can do anything the logged-in user can do on any page it navigates to: create API

what pages and sessions are open when using Claude in Chrome for automation tasks.
Claude does not have elevated access beyond what the browser session already has,
but it does have FULL access to everything that session can reach.

Example: Claude created a new Gitea API token with issue write permissions by
navigating to the token settings page where the user was already logged in. No
privilege escalation occurred -- Claude simply used the existing authenticated session.

2026-03-15 | Session 17 (Promtail install)
PI 3B+ IS 64-BIT (ARM64) -- DO NOT ASSUME 32-BIT
The Raspberry Pi 3B+ has a Cortex-A53 processor (ARMv8), which is 64-bit capable.
When running Raspberry Pi OS Lite 64-bit, the kernel and userspace are both 64-bit.
Always use linux-arm64 binaries, not linux-arm (armhf/32-bit) binaries. The 32-bit
binary will fail at execution with "required file not found" because the 32-bit ELF
interpreter (/lib/ld-linux-armhf.so.3) is not installed on the 64-bit OS.

2026-03-20 | v3.3.57-v3.3.59
SESSION IDLE TIMEOUT — THREE COMPOUNDING BUGS
The session idle timeout feature had three separate bugs that had to be fixed in sequence:
1. /api/session/status and /api/session/ping had @auth.login_required. When the session
   expired, these returned a redirect instead of JSON. The client JS catch() swallowed
   the error silently and the redirect to /login never fired.
2. /api/session/status and /api/session/ping were missing from _POLLING_PATHS, so the
   30-second checkRemaining() call was resetting last_activity every tick via
   update_last_activity() before_request hook.
3. Every auto-refreshing page (dashboard every 10s, logs, metrics) called API endpoints
   not in _POLLING_PATHS, resetting last_activity continuously. The blocklist approach
   is fundamentally fragile — any new polling endpoint not added to the list breaks
   idle timeout silently. Fix: removed update_last_activity() from before_request
   entirely. last_activity is now updated ONLY by /api/session/ping, which is triggered
   by real user events (mousedown, keydown, touchstart, scroll, wheel).
After installing the fix, users must log out and back in to get a clean session —
the old cookie carries a stale last_activity timestamp that the new code reads correctly
but which may be arbitrarily old.

DELETE OUTPUT ZIP BEFORE REBUILDING -- zip UPDATES NOT REPLACES
The zip command updates an existing archive rather than replacing it. If a package
zip is rebuilt without first deleting the old file, files from the previous build
remain in the zip alongside the new files. This causes patcher's _find_in_zip()

  cd /package_dir && zip -r ../output.zip .
This is in the process doc packaging rules but worth repeating: never zip without
first confirming the output file does not already exist.

2026-03-27 | Session 36
PACKAGES WITH run_sql MUST ALSO REPLACE MIGRATIONS.md ON DISK
When a package includes a run_sql action, it must also include a replace action
for MIGRATIONS.md with the new entry added. Without the replace action, patcher
executes the SQL against the local DB but never writes the updated MIGRATIONS.md
to disk. When build_release.sh runs to produce the next release zip, it copies
MIGRATIONS.md from disk — which still contains the old content with no entry for
that version. The release zip ships a blank migrations file, apply_migrations()
finds no entries to apply, and skip-version upgraders never get the schema change.

Rule: every package with a run_sql action requires TWO corresponding changes:
  1. The same SQL added to MIGRATIONS.md under the new version heading
  2. A { "action": "replace", "file": "MIGRATIONS.md" } entry in the manifest

The existing process doc rule ("run_sql in a package → add SQL to MIGRATIONS.md")
was necessary but not sufficient. The replace action is equally required.
Root cause discovered during skip-version migration test, Session 36.

CLOSEOUT PACKAGES REQUIRE A MANIFEST JUST LIKE ANY OTHER PACKAGE
All packages installed via the Package Installer require a manifest.json at the
zip root — including doc-only closeout packages. The manifest must have an "actions"
list (even if it only contains replace actions for dev_docs files) and no restart
action is needed for doc-only packages. Without a manifest.json, the patcher rejects
the zip entirely.

Correct closeout manifest example:
  {
    "version": "X.Y.Z",
    "description": "Session N closeout: summary, CurrentTask, context doc.",
    "session": N,
    "actions": [
      { "action": "replace", "file": "dev_docs/ChatSummary_N.txt" },
      { "action": "replace", "file": "dev_docs/CurrentTask.txt" },
      { "action": "replace", "file": "dev_docs/NetWatch_AI_Context_latest.txt" }
    ]
  }

No "restart" action needed — patcher skips it and no services are touched.
