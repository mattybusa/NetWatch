2026-02-28 | pre-v2.0
CONTEXT COMPACTION LOSES GENERATED CODE
Claude's context window has a hard limit. During long code generation sessions,
earlier content is silently dropped. If the session was generating a large codebase,
partial or complete regeneration may be required after compaction. Always save working
code to disk before continuing a long session. For large code sessions, present files
for download as soon as they are generated and confirm the user has saved them before
the session grows further.

2026-02-28 | pre-v2.0
SCP FROM WINDOWS — MUST USE SEPARATE POWERSHELL WINDOW, NOT INSIDE SSH SESSION
SCP commands cannot be run from inside an active SSH session. Open a separate
PowerShell window (not the SSH shell) and run SCP from there. Running SCP inside SSH
produces an error and the file transfer will not complete.

2026-03-01 | v2.4
DUPLICATE FLASK ROUTES AFTER PATCHING webapp.py
Incremental patching of webapp.py can introduce duplicate @app.route decorators.
Flask silently ignores the second definition, causing old route behavior to persist.
After patching webapp.py, always check for duplicate @app.route decorators.
The patcher now performs a preflight check and refuses to apply packages with
duplicate routes.

2026-03-01 | v2.3-ish
SCP FILE TRANSFER NOT TAKING EFFECT — PYCOM CACHE
After SCP-ing a Python file to the Pi, the old behavior persists because Python caches
compiled .pyc files. Delete the __pycache__ directory or restart the service to force
reload. Under gunicorn with preload_app=True, a full service restart is required.

2026-03-01 | v2.3–v3.2.37
CONFIG EDITOR SAVE: THREE SEPARATE ROOT CAUSES
The config editor save bug had three independent root causes: (1) UTF-8 encoding not
specified on file open/write, (2) service restart called before file write flushed to
disk, (3) path resolution using __file__ instead of os.path.expanduser(). All three
must be fixed together — fixing one at a time leaves the others active. Final fix
shipped v3.2.37.

2026-03-02 | v3.2.50f
SQLITE3 c.execute() CRASHES ON MULTI-STATEMENT STRINGS
Python's sqlite3 module raises ProgrammingError if a single c.execute() call contains
multiple SQL statements separated by semicolons. This is silent at schema design time
but crashes the monitor on startup. Always use separate c.execute() calls for each
statement in init_db() and any other schema initialization code.

2026-03-02 | general
CHART.JS autoSkip:false AND BLANK-STRING makeLabels() ARE A PAIRED DESIGN
The health chart label architecture uses autoSkip:false with a makeLabels() function
that returns blank strings for non-display positions. These two settings are coupled —
changing autoSkip to true while keeping blank-string labels causes most labels to
disappear. Changing makeLabels() without setting autoSkip:false causes label crowding.
Do not change one without reviewing the other.

alerts.py is imported by both netwatch-monitor (main monitoring loop) and netwatch-web

2026-03-07 | v3.2.55–v3.2.57

2026-03-07, 2026-03-08 | v3.2.75, v3.2.82, v3.2.85 (Session 2); general (Session 3+)
DO NOT DIAGNOSE FROM SURFACE CODE READS
Multiple incidents were caused by concluding something was broken or missing after
a shallow grep or partial read. Always read the full relevant code before forming
a conclusion. If the user says it works, it works. Find why before saying otherwise.
Canonical incidents: v3.2.75 (broke working password reset), v3.2.82 (misdiagnosed
24H data gap as clock skew — was real monitor downtime), v3.2.85 (rewrote working
datetime queries), Session 4 SMS change detection (dismissed as not implemented when
it was fully working — user had screenshots proving it), Session 1 1H chart empty
(diagnosed as clock drift — wrong; real cause was monitor crash from multi-statement
c.execute() in v3.2.49).
Rule: simulate before shipping. If the user says something works, find the actual cause
before changing it. When user says "it worked until we changed X," diff that specific
change against its prior version before hypothesizing external causes.

2026-03-11 | v3.3.2–v3.3.4
PURE ASCII ONLY IN ALL .sh FILES
Shell scripts containing Unicode box-drawing or symbol characters (e.g. = - + arrows)
cause bash syntax errors on the Pi when transferred via SCP or Dev Docs upload,
even if the file is valid UTF-8. The Pi's bash parser chokes on multi-byte characters
in certain contexts. Rule: ALL .sh files must use pure ASCII only. No decorative
box-drawing, no Unicode arrows, no Unicode checkmarks. Use +, -, =, and plain text
equivalents instead.

2026-03-11 | v3.3.2–v3.3.4
NEVER USE BASH HEREDOCS IN SHELL SCRIPTS
Bash heredocs (<< LABEL) fail silently on the Pi in certain shell environments. The
rollback.sh DB restore used python3 - << PYEOF which ran without error but did nothing.
Use printf to build temp Python files instead, passing content via a temp file argument.
This was the root cause of the v3.3.2 DB restore failure.

ROLLBACK RESTORES .sh FILES FROM SNAPSHOT
If you fix rollback.sh via a package and then immediately run rollback, the old broken

2026-03-13 | v3.3.19
JINJA2 NESTED BLOCKS CAUSE DOUBLE RENDERING
If {% block scripts %} is opened inside {% block content %} in a child template, but
base.html defines them as separate sequential blocks, Jinja2 renders the scripts content
TWICE -- once inline within content, and again at the standalone {% block scripts %}
location. This produces duplicate JS declarations and a SyntaxError that silently kills
all JS on the page. Always verify block structure: each block in a child template must be
a top-level sibling, not nested inside another block, unless base.html also nests them
the same way.

2026-03-13 | v3.3.19-v3.3.21
CSS CUSTOM PROPERTIES: body BEATS :root FOR INHERITANCE
CSS custom properties (variables) resolve by DOM inheritance, not just specificity. If
body.theme-* sets --bg-card on body, that value wins over :root { --bg-card } because body
is a closer ancestor to all elements than :root. Setting variables on :root does NOT
override variables set on body. To override body.theme-* variables: use a selector with
equal or higher specificity (e.g. body[class]) placed AFTER themes.css in the <head> so
source order breaks the tie.

2026-03-13 | v3.3.25
SERVICE ACCOUNT HOME DIRECTORY TRAVERSAL -- /home/mboynton MUST BE 751
When a systemd service runs as a different user (e.g. netwatch-svc) with WorkingDirectory
inside /home/mboynton/, the Pi's default home dir permissions of 700 (drwx------) will
cause the service to fail at startup with status=200/CHDIR (Permission denied). The
service account needs execute permission on the parent directory to traverse into it.
Fix: sudo chmod 751 /home/mboynton. The 751 gives others execute-only (no directory
listing) while allowing traversal. This is a one-time manual step -- it does not ride
in a package.

2026-03-13 | v3.3.25
PATCHER LEAVES TEMP .pyc FILES IN /tmp/__pycache__/ -- NEVER CLEANED UP
patcher.py performs a syntax check by compiling each Python file to a temp .pyc in
/tmp/__pycache__/. It never cleans up these files. They accumulate across every package
install and are owned by whatever user ran the install. If the directory exists with
wrong ownership/permissions when a new user (e.g. netwatch-svc via web service) tries
to write there, the patcher reports a misleading "syntax error in filename.py" when the
real error is a permission failure on the temp file. Workaround: sudo chmod 777
/tmp/__pycache__. Real fix (backlog): use tempfile.mkstemp() in patcher.py and clean
up after the syntax check completes or fails.

2026-03-13 | v3.3.25
SERVICE ACCOUNT WITH --no-create-home: expanduser() BREAKS -- USE __file__ PATHS
When a Python script runs as a system account created with --home-dir /nonexistent
--no-create-home, os.path.expanduser("~") resolves to /nonexistent. Any path built
with expanduser() will fail when the code tries to create or access that path.
All Python files that run as the service account (main.py, theme_manager.py) must use
os.path.dirname(os.path.abspath(__file__)) to build paths relative to the script
location instead. Shell scripts that run as mboynton over SSH are unaffected -- $HOME
resolves correctly for the interactive user.

GOTCHA: PACKAGE ZIP STRUCTURE MUST MIRROR TARGET PATH STRUCTURE
Session 10 | 2026-03-13

Files in a package zip must be placed inside a subdirectory matching their target
path prefix. For example, dev_docs files must be inside a dev_docs/ folder in the
zip, not at the zip root. The patcher validation step checks zip structure and will
hang on "validating" indefinitely if files are at the wrong level.

Manifest.json always sits at the zip root. Everything else mirrors its target path.

Correct structure:
  manifest.json          (root)
  dev_docs/
    ChatSummary_10.txt
    Gotchas.txt
    ...
  templates/
    some_template.html
  static/
    some_file.js

Wrong structure (causes hang on "validating"):
  manifest.json
  ChatSummary_10.txt     (should be inside dev_docs/)
  Gotchas.txt
  ...

The _find_in_zip function matches by basename, so the patcher CAN find files
at the wrong level -- but an earlier validation step rejects the package before
_find_in_zip is ever reached. Always match zip structure to target path structure.

GIT ON A SERVICE-ACCOUNT-OWNED DIRECTORY
-----------------------------------------
~/netwatch is owned by netwatch-svc:netwatch-svc with permissions drwxr-x---.
Running git init as mboynton fails with "Permission denied" even after adding
mboynton to the netwatch-svc group, because the directory itself lacks group
write permission.

Fix requires two steps:
1. sudo usermod -a -G netwatch-svc mboynton (then log out and back in)
2. sudo chmod g+w ~/netwatch

Also required:
  git config --global --add safe.directory /home/mboynton/netwatch
  (git refuses to operate on directories owned by a different user without this)

GIT FIRST PUSH WITH INITIALIZED REMOTE
----------------------------------------
When you create a repo on Gitea with "Initialize repository" checked (adds
README and LICENSE), and then try to push a local repo for the first time,
git rejects the push because the remote has commits the local doesn't have.

Fix:
  git config --global pull.rebase false
  git pull origin main --allow-unrelated-histories
  (resolve any conflicts, commit the merge)
  git push -u origin main

GITEA INITIAL SETUP WIZARD REWRITES app.ini
---------------------------------------------
The Gitea setup wizard rewrites app.ini without section headers, breaking
the config. After running the wizard:
1. Verify /etc/gitea/app.ini still has [server], [database], etc. headers
2. If headers are missing, rewrite the file manually with correct structure
3. Set INSTALL_LOCK = true in [security] to prevent wizard from reappearing
4. Permissions must be temporarily relaxed for the wizard to write the file:
   sudo chmod 777 /etc/gitea && sudo chmod 666 /etc/gitea/app.ini
   Lock back down after install: sudo chmod 750 /etc/gitea && sudo chmod 640 /etc/gitea/app.ini

NETPLAN CONNECTION NAME ON RASPBERRY PI OS LITE 64-BIT
--------------------------------------------------------
Newer Raspberry Pi OS Lite 64-bit uses NetworkManager with netplan, not dhcpcd.
dhcpcd.service does not exist. nmcli is the correct tool.
Connection name is NOT "Wired connection 1" -- check with: sudo nmcli con show
Actual name on a fresh Pi is typically "netplan-eth0".
Static IP command:
  sudo nmcli con mod "netplan-eth0" ipv4.addresses X.X.X.X/24 \
    ipv4.gateway X.X.X.X ipv4.dns 8.8.8.8 ipv4.method manual
  sudo nmcli con up "netplan-eth0"

GITEA BUILT-IN SSH VS SYSTEM SSH PORT CONFLICT
------------------------------------------------
Gitea's built-in SSH server and the system SSH daemon both want port 22.
Setting SSH_PORT = 2222 in app.ini does not automatically make Gitea's
built-in SSH server bind to 2222 -- it may simply not start.
Workaround: use HTTPS with stored credentials (git credential.helper store).
This is fully functional and passwordless after first push.
Gitea SSH passthrough (using system SSH to route git connections to Gitea)
is an alternative but requires additional configuration.

OPENSSL SERIAL FILE PERMISSIONS
---------------------------------
When signing a certificate with openssl x509, if the CA serial file (.srl)
is not writable by the current user, the signing produces a 0-byte output
file with no clear error message -- just a permissions warning in stderr.
Fix: sudo chmod 664 ~/netwatch/certs/ca/netwatch-ca.srl before signing.
