WHAT WAS ACCOMPLISHED

ARC 1 -- Observability stack architecture discussion

- Clarified the relationship between OpenTelemetry, Prometheus, Loki, Node
  Exporter, Promtail, and Grafana. Key decisions confirmed:
  - OTEL is the standard (format/API), not an implementation
  - Prometheus is a metrics storage/scraping engine (OTEL-compatible backend)
  - Loki is a log storage/query engine (Grafana Labs)
  - Node Exporter exposes system metrics for Prometheus to scrape (pull model)
  - Promtail ships logs from each Pi to Loki (push model)
  - Grafana is the dashboard layer on top of both
  - The OTEL Collector was discussed as an optional middleware layer --
    decided to defer it; Promtail + Node Exporter is the simpler correct path
    for a two-Pi home LAN setup

- mBase alignment confirmed: the commitment is to the FORMAT (structlog JSON,
  OTEL-compatible shape) in the app code. The plumbing (Promtail vs OTEL
  Collector) is infrastructure outside the app and can be swapped later without
  touching app code. This is consistent with "mBase CONNECTS, doesn't DUPLICATE."

- Correct installation order established:
  1. Rework NetWatch logging to structlog JSON first (DEFERRED to next session)
  2. Then install Promtail to ship structured logs to Loki
  3. NOT the reverse -- installing Promtail before structlog would ingest
     unstructured text that is hard to query and would need redoing anyway

- DECISION: Host for observability stack -- Pi 4 8GB (previously reserved for
  future project) reassigned to ops-pi role. Reasoning: Pi 3B+ (1GB RAM) is
  too tight for Gitea + Prometheus + Loki + Grafana simultaneously.

WHAT WAS ATTEMPTED BUT FAILED
------------------------------
- Pi 3B+ SD card swap to Pi 4 considered: decided against. Fresh install
  safer and cleaner given we were installing a whole new stack anyway.
  Model-specific firmware in boot partition makes 3B+ -> Pi 4 migration
  unreliable.

- Gitea 1.22.6 installed initially: crashed immediately with migration error.
  SSD database was created by Gitea 1.25.5 (migration version 323). 1.22.6
  only supports up to version 299. Replaced with 1.25.5 which matched.

- Gitea startup failed with permission denied on app.ini: Gitea needs write
  access to app.ini on first startup to write its internal token. Fixed by
  setting app.ini ownership to root:gitea with 660 permissions before first
  start, then locking down to 640 after token written.

- Gitea startup failed with git not found: git package not installed on fresh
  Pi. Fixed with sudo apt install git -y.

- Loki config missing delete_request_store: Loki 3.5 requires
  delete_request_store to be set in the compactor block when retention is
  enabled. Config error on first start. Fixed by adding
  delete_request_store: filesystem to compactor config.

- Prometheus 3.5.1 tarball did not contain consoles/ or console_libraries/
  directories: these were removed in Prometheus 3.x. cp commands failed.
  Not needed for our setup -- binaries and config are sufficient.


DECISIONS MADE
--------------
- ops-pi is now the canonical name for 192.168.100.230. All future references
  should use ops-pi.home, not gitea-pi.home.

- Pi 4 8GB is the correct hardware for the ops-pi role. 8GB gives ample
  headroom for Gitea + Prometheus + Loki + Grafana + future services.

- Promtail deferred until after NetWatch structlog rework. Installing Promtail
  before structured logging is in place would result in unstructured log
  ingestion that would need to be redone.

- Grafana on port 3001, Gitea on port 3000. Both on ops-pi.home. No HTTPS
  on Grafana/Prometheus/Loki yet -- deferred, HTTP on LAN acceptable for now.
  TLS for these services is a future backlog item.

- OTEL Collector deferred indefinitely for current two-Pi setup. Will revisit
  when there is a concrete reason to need routing flexibility (e.g. swapping
  backends or multiple destinations). The structlog JSON format in NetWatch
  keeps the door open without adding complexity now.

- Node Exporter is the correct tool for system metrics (CPU, memory, disk,
  network). It predates OTEL but is the de facto standard and is widely
  supported. Using it does not conflict with the OTEL standards-alignment goal.

- End-of-session context doc updated at v3.3.36 (no version bump -- no
  NetWatch code changes this session).

NEW GOTCHAS ADDED
-----------------
Appended directly to Gotchas.txt during session per process:

- LOKI 3.X REQUIRES delete_request_store WHEN RETENTION ENABLED (new) --
  Loki 3.x compactor block requires delete_request_store to be set when
  retention_enabled: true. Config validation error on startup if missing.

- GITEA REQUIRES WRITE ACCESS TO app.ini ON FIRST STARTUP (new) --
  Gitea writes its internal token to app.ini on first run. Service crashes
  if app.ini is not writable by the gitea user at that point. Set 660
  before first start, lock to 640 after.

- GITEA DATABASE MIGRATION VERSION MUST MATCH BINARY VERSION (new) --
  Gitea refuses to start if the database migration version is newer than
  the binary supports. The version that created the database must be matched
  exactly or exceeded. Check migration version in startup logs if Gitea
  crashes immediately after install.
