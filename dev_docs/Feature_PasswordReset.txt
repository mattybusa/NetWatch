Feature_PasswordReset.txt
=========================
FEATURE/BUG HEADER
------------------
Name: Password Reset via Email + SMS (including self-introduced regression)
Type: Feature + Self-introduced regression arc
Date worked on: 2026-03-07
Sessions: Session 2
Final outcome: Complete (v3.2.77 confirmed working by user)


BACKGROUND
----------
Password reset via email was already built in a prior session and confirmed working.
The request in this session was to verify the existing implementation, confirm it
sends to SMS when the account has a verified phone, and close any gaps. The user also
wanted to understand if the existing code could lock out a user if the reset email
failed to deliver.


WHAT WAS TRIED AND FAILED
--------------------------
Google Voice SMS gateway: User suggested number@txt.voice.google.com as a gateway.
Researched — not a static inbound gateway. The format is dynamically per-conversation
with a random token. Not added to carrier list. No mention in UI.

Self-introduced regression (v3.2.75–v3.2.77) — approximately 1 hour wasted:

During the v3.2.74 work, an incorrect code read concluded that request_password_reset()
in auth.py still had password_hash=? in its UPDATE statement (i.e. still overwriting
the real password, a bug that was supposed to have been fixed in v3.2.69/70).

v3.2.75 was built to: (1) add SMS to password reset, (2) "fix" the password_hash
overwrite. But the overwrite had already been fixed — it was not in pi_current. The
"fix" was applied to a problem that did not exist.

After v3.2.75 installed: user reported real password login was now forcing a password
change. This was a new bug introduced by v3.2.75's other change. v3.2.76 was built
to fix it (clear must_change_pass on real-password login).

After v3.2.76 installed: user reported temp password no longer worked. Root cause:
v3.2.75 removed the password_hash write from request_password_reset. Since
authenticate() previously only checked password_hash, and the temp hash was now ONLY
in reset_token_hash, the temp password had no path to succeed. v3.2.77 was built to
add reset_token_hash checking to authenticate().

User pushed back at each step and was correct. The session Gotchas log already
contained a "DO NOT DIAGNOSE FROM SURFACE CODE READS" entry from prior incidents.
This session added another incident to the same pattern (see Gotchas_Append_Session2.txt
for amendment entry).

Full arc of what actually happened:
- v3.2.69/70 (prior session): request_password_reset() wrote temp hash to
  reset_token_hash ONLY. authenticate() checked both password_hash AND reset_token_hash.
  Both tested and confirmed working.
- v3.2.75 (this session): Misread prior-session auth.py as still having the dual write.
  "Fixed" it by removing password_hash=? — but it was already gone. Net effect on
  password_hash: none. But this also added SMS send and must_change_pass changes.
- v3.2.76: Added must_change_pass=0 clear on real-password login (legitimately needed).
- v3.2.77: Added reset_token_hash check to authenticate() again (redundant with v3.2.70
  but necessary because v3.2.75 had disrupted the relationship between the columns
  in ways that made v3.2.77 the correct completion).

Net result of v3.2.75–v3.2.77: functionally equivalent to v3.2.70, with the addition
of SMS send, delivery logging for both channels, and correct must_change_pass handling.


WHAT WORKED
-----------
Package v3.2.68: request_password_reset() updated to also look up verified phone/SMS
address and return both email and SMS info. webapp.py forgot_password route updated
to send to both email and SMS when both are verified. Security log detail updated to
show which channels were used (e.g. "Reset code sent via email (mat***) + SMS (703***)")

Package v3.2.69: Fixed bug discovered during testing — request_password_reset was
overwriting password_hash with temp hash. Real password became invalid the moment
reset was requested. Fix: write temp hash to reset_token_hash only. authenticate()
updated to also check reset_token_hash.

Package v3.2.70: Fixed must_change_pass behavior — if user requested a reset then
logged in with their real password, should not be forced to change. Fix: on successful
real-password login, clear must_change_pass if it was set by a pending reset.

(Note: v3.2.75–v3.2.77 are the regression arc documented above. v3.2.77 is the
current confirmed-working state.)

Final confirmed behavior (v3.2.77):
- request_password_reset() writes to reset_token_hash only — password_hash untouched
- authenticate() tries both password_hash and reset_token_hash independently
- Real password login: token cleared, must_change_pass=0, no forced change
- Temp password login: token cleared, must_change_pass stays 1, forced change screen
- Token expires after 15 minutes
- must_change_pass re-read from DB on every request (session value is not authoritative)
- Reset sends to both email and SMS when both contact methods are verified
- Delivery history logs both sends with correct subscriber_id

Files changed: auth.py (authenticate, request_password_reset), webapp.py (forgot_password
route), preferences.html (SMS change detection — v3.2.71, separate package)

User confirmed v3.2.70 working (both temp and real password tested), and later
confirmed v3.2.77 working after the regression arc resolved.

[UNVERIFIED — SMS path specifically: only mboynton's account has both email and SMS
verified. The SMS send path in the reset was not separately confirmed vs. email-only
accounts, though the security log detail string was confirmed to show both channels
when both are present.]


KNOWN LIMITATIONS / DEFERRED ITEMS
------------------------------------
SMS gateway limitation: email-to-SMS carrier gateway only. No carrier-agnostic API
selected yet (user declined Textbelt for now, continuing to research). Google Voice
gateway does not work.

Email-to-SMS gateways require knowing the carrier. Carrier selection is a field in
My Account. If the carrier is wrong, SMS will silently fail to deliver.
