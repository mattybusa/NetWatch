- v3.3.22 — Interface-bound network checks (see Feature_NetworkInterfaceChecks.txt)
  Added LAN_INTERFACE and WIFI_INTERFACE config keys. ping() now accepts an optional
  interface parameter and passes -I <iface> to the OS. New _bound_ping() helper does

  Confirmed working by user.

- v3.3.24 — Config editor NEEDS SETUP badge for newly added keys
  (see Feature_FileUploaderAndConfigEditor.txt — revision block appended)
  When the validator auto-adds a new key with a valid default (e.g. SNAPSHOT_RETENTION),
  it now shows a NEEDS SETUP badge so the user can confirm the value. Previously the
  badge only fired for known placeholder values; valid defaults slipped through silently.
  Two-path detection: Case 1 (placeholder values in UNCONFIGURED_VALUES), Case 2
  (undismissed config_notification in DB). Saving the Config Editor dismisses
  notifications for saved keys. Also fixed: optional keys with blank values
  (LAN_INTERFACE, WIFI_GATEWAY, WIFI_INTERFACE) no longer trigger spurious NEEDS SETUP
  badges. Verified with test keys (v3.3.25); test keys removed (v3.3.26); user rolled
  back to v3.3.24 (test packages not needed on Pi).

DECISIONS MADE
--------------
- Interface-bound pings: user's .100 and .110 subnets share the same VLAN. Without
  interface binding, unbound pings could route out either NIC, making WiFi/LAN checks
  unreliable. Decision: force pings via -I <iface>; WAN always unbound since it tests
  internet reachability, not a specific NIC.

- Optional keys / blank-is-valid: LAN_INTERFACE, WIFI_INTERFACE, WIFI_GATEWAY have
  required=False and default="". Blank is a valid configured state for these keys.
  They should never trigger NEEDS SETUP from the placeholder path. Only trigger via
  Case 2 (new-key notification) if the validator auto-added them.

- Test key rollback: user chose to roll back rather than install cleanup package v3.3.26.
  Pi is at v3.3.24. Orphan test key lines may remain in config.py on the Pi — harmless,
  validator ignores unknown keys.
