## ⚠️ CRITICAL: How to Work With This Project

### Before proposing ANY fix or change:
1. **Read everything first.** Read all uploaded transcripts, all referenced files, all context. Do not assume. Do not guess. If something seems like a bug, verify it exists in the actual current code before touching it.
2. **Do not fix what isn't broken.** Multiple hours were lost fixing auth, database queries, and password reset flows that were working correctly. The v3.2.75 incident (broke working password reset by "fixing" it) and the v3.2.85 incident (unnecessarily rewrote datetime queries) are the canonical examples.
3. **When the user says something works or is broken — believe them.** Do not tell the user they're mistaken, that they're "not seeing it correctly," or that the logs say otherwise. The user has been right every single time this happened. Acknowledge, investigate, find the real cause.
4. **Simulate and verify before shipping.** If a fix can be simulated in Python or traced through the code logically, do it before building a package. Never ship a package based on a theory.
5. **Do not introduce scope creep.** Fix what was asked. Note other things you spotted, but ask before touching them.

**User profile:**
- Technically proficient, logical thinker, not a programmer, Linux skills are limited
- Cannot always tell what a code block does or how to adapt it — be explicit
- Good at describing what they want in plain language
- Questions and action items must be clearly visible — don't bury them in prose
- Always flag security risks and realistic mitigations

### database.py datetime queries — critical rule
- All timestamps stored as UTC via `datetime.utcnow().strftime("%Y-%m-%d %H:%M:%S.%f")`
- All WHERE clauses use `datetime('now', '-N hours')` — SQLite's `datetime('now')` is UTC on the Pi and correctly matches stored timestamps
- **Never replace `datetime('now')` with Python-computed cutoffs** — was tried in v3.2.85 and reverted in v3.2.86; the original SQLite approach is correct
- `timedelta` is not imported in database.py — don't add it unless genuinely needed for non-query logic

## Custom Theme System (v3.3.13–v3.3.21)

Custom themes extend the built-in system themes (dark-blue, dark-green, light, high-contrast).
They are stored as `.nwtheme` files in `~/netwatch/themes/`.

**File format:** JSON with two top-level keys:
- `"color_scheme"`: dict of 25 CSS variables (REQUIRED_COLOR_KEYS in theme_manager.py)
- `"layout"`: dict of 3 CSS variables (REQUIRED_LAYOUT_KEYS in theme_manager.py)

**Key module:** `theme_manager.py`
- `save_theme(name, data)` — validate and write .nwtheme file
- `load_theme(name)` — read .nwtheme file
- `list_themes()` — list all themes with enabled/disabled status
- `delete_theme(name)` — remove .nwtheme and .disabled sidecar
- `generate_css(theme, part)` — produce CSS block using `body[class]` selector
- `toggle_theme(name)` — create/remove .disabled sidecar file

**CSS injection:** `base.html` injects a `<style id="custom-theme-css">` block AFTER
`themes.css` in `<head>`. The system theme class stays on `<body>` always. The custom
variables override via `body[class] { ... }` — equal specificity to `body.theme-*`,
wins on source order. No class switching = no flash on load.

**User pref keys:** `custom_color_scheme` and `custom_layout` in `user_prefs` table.
Setting to `None` via `set_user_pref(id, key, None)` deletes the row (reverts to system default).

**Permissions:**
- Admin only: import, export, delete, enable/disable
- All users: select from enabled custom themes in appearance popover and preferences page

**Disabled themes:** `.disabled` sidecar file alongside the `.nwtheme` file.
Disabled themes are hidden from user selectors but remain on disk.

## alerts.py — Key Facts

- `alerts.py` is imported by **both** `monitor.py` and `webapp.py`
- Test emails triggered from the web call `alerts.send_test()` directly from the web service
- **Any change to `alerts.py` must restart both services**
- Email via Gmail SMTP using `GMAIL_USER` + `GMAIL_APP_PASSWORD` from config.py
- `GMAIL_USER` is the sending credential only — doesn't need to match any user account email
- Owner subscriber matched by `owner_username` field in `alert_subscribers`, not email
- MIMEText must specify `"utf-8"` charset: `MIMEText(body, "html", "utf-8")`
- Gmail blocks `data:image/png;base64,...` src attributes — use emoji instead

### Alert system master lists (in alert_subscribers.py)
- **`ALERT_TYPES`** — the single source of truth for every alert type in the system. Controls what appears in subscriber preferences, delivery history filters, role defaults, and the DB seed. Adding a new alert type anywhere requires adding it here first.
- **`CARRIERS`** — master list of SMS carrier gateway domains. Adding a new carrier requires adding it here; `CARRIER_BY_ID` is derived from this list automatically.

## Metrics Page — Charts (v3.2.78+)

### Architecture
- All 4 charts use shared Chart.js plugins registered once: `CrosshairPlugin`, `ZoomDragPlugin`
- Data flow: `loadData(hours, startIso?, endIso?)` → fetch API → `bucketAvg()` → chart update
- Auto-refresh pauses while zoomed (`isZoomed` flag)

### Key functions
```javascript
bucketAvg(arr, fields, numBuckets, startMs, endMs)  // time-anchored bucket averaging
makeBucketLabels(buckets, hours)    // sparse x-axis labels
makeBucketRawLabels(buckets)        // full "Avg: Mar 7 06:35 AM – 06:50 AM" tooltip strings
numBuckets(hours)                   // 1H→60, 6H→72, 24H→96, 7D→168, 30D→240
loadData(hours, startIso, endIso)   // main data loader
```

### Bucket window anchoring — critical rule
- `startMs` is always `Date.now() - hours*3600*1000` (full requested range)
- `endMs` is always `Date.now()`
- **Never derive startMs from oldest data point** — was tried in v3.2.82/83, reverted in v3.2.86
- Gaps in data (monitor restart, Pi reboot) correctly appear as breaks in the line
- The old `sample()` function is gone — do not reintroduce it

### chart._rawLabels
- Stored per chart: full timestamp string per bucket point, used by CrosshairPlugin tooltip
- Set alongside chart data update: `chart._rawLabels = makeBucketRawLabels(buckets)`

### Zoom
- Drag on any chart to zoom — ZoomDragPlugin computes ISO timestamps from drag bounds
- Calls `loadData(hours, startIso, endIso)` with explicit window
- Reset Zoom button appears in toolbar while zoomed

## How to Deploy Updates to NetWatch

There are **two methods** for deploying code changes to the Pi. Claude builds the package;
the user installs it via the NetWatch web UI. No CLI needed.

---

### Method A — Package Installer (preferred for any change)

**When to use:** Any time you are changing one or more files. This is the standard method.
It takes a snapshot before touching anything, so rollback is always available.

**UI path:** Admin → Package Installer (at `/admin/patch`)

**What Claude does — step by step:**

1. Edit the file(s) in the working directory
2. Bump the version number (e.g. `3.2.92` → `3.2.93`)
3. Build a zip containing:
   - `manifest.json` at the **root** of the zip
   - All changed files at the **root** of the zip (patcher finds them by basename)
   - Use `manifest.json` file entries with full relative paths (`templates/foo.html`, `static/bar.css`, `dev_docs/file.txt`)
4. Provide the zip for download

**CRITICAL — Zip must be built from inside the package directory:**
```bash
cd /package_dir && zip -r output.zip .
```
Never zip a folder from outside — files must be at the zip root, not inside a subdirectory.

**What the user does:**

1. Download the zip from Claude
2. Open NetWatch → Admin → Package Installer
3. Upload the zip — the UI shows a preview of every action before applying
4. Click Apply
5. Wait for the restart confirmation

---

### Manifest format

```json
{
  "version": "3.2.93",
  "description": "Brief description of what this package does.",
  "min_version": "3.2.92",
  "actions": [
    { "action": "replace", "file": "webapp.py" },
    { "action": "replace", "file": "templates/patch_manager.html" },
    { "action": "replace", "file": "static/themes.css" },
    { "action": "replace", "file": "dev_docs/NetWatch_AI_Context_latest.txt" },
    { "action": "restart", "services": ["monitor", "web"] }
  ]
}
```

**Manifest rules (all confirmed from source):**

- `version` — new version string; written to the Pi's VERSION file on success
- `description` — shown in changelog; be specific
- `min_version` — optional; prevents install on too-old versions
- `actions` list is executed **in order**; a failed action stops execution and skips remaining steps
- `restart` action should always be **last**
- **Always include the updated context doc** as a `replace` action targeting `dev_docs/NetWatch_AI_Context_latest.txt`

---

### File path rules in manifest

`_resolve_path()` in patcher.py determines where each file lands. The rules are:

| File path in manifest | Where it is written on Pi |
|---|---|
| `webapp.py` | `~/netwatch/webapp.py` |
| `main.py` | `~/netwatch/main.py` |
| `templates/index.html` | `~/netwatch/templates/index.html` |
| `index.html` (no prefix) | `~/netwatch/templates/index.html` (auto-placed by `.html` extension) |
| `static/themes.css` | `~/netwatch/static/themes.css` |
| `themes.css` (no prefix) | `~/netwatch/static/themes.css` (auto-placed by `.css` extension) |
| `dev_docs/filename.txt` | `~/netwatch/dev_docs/filename.txt` (v3.2.90+) |

**Best practice:** Always use the full relative path with subdirectory prefix (`templates/`, `static/`, `dev_docs/`)
for HTML, CSS, JS, and dev doc files. This is unambiguous and immune to future logic changes.

`_find_in_zip()` matches by **basename** anywhere in the zip, so the file can sit at the
zip root or in a subfolder — patcher finds it either way.

---

### Service restart rules

`_queue_restart()` in patcher.py calls `updater.get_service_for_file()` for each replaced file,
then also honours explicit `restart` actions. The service map (from `updater.py`) is:

| File | Service(s) restarted |
|---|---|
| `main.py`, `monitor.py`, `network.py`, `relay.py`, `button.py` | `netwatch-monitor` only |
| `webapp.py`, `updater.py`, `configeditor.py`, `patcher.py`, `certmanager.py`, `config_validator.py`, `security_log.py` | `netwatch-web` only |
| `alerts.py`, `alert_subscribers.py`, `database.py`, `auth.py`, `config.py` | **both** services |
| `.html`, `.css`, `.js` files | `netwatch-web` only |

**Always include an explicit `restart` action** — do not rely on auto-detection alone.
The safe default is `"services": ["monitor", "web"]` (restarts both).

**How web restart works (critical):** `netwatch-web` is restarted with a **2-second background delay**
(`sleep 2 && sudo systemctl restart netwatch-web`) so the HTTP response fully completes first.
The user will see a brief disconnect (~5 seconds) and then the page reloads. This is normal.

**How monitor restart works:** `netwatch-monitor` is restarted synchronously (no delay).
The network monitoring loop pauses briefly during restart.

**gunicorn `preload_app = True`:** The app loads once in the master process before forking workers.
`init()` runs once, not once per worker. A full service restart (`systemctl restart`) is required
to pick up code changes — a reload signal is not sufficient.

### All available action types

| Action | Purpose |
|---|---|
| `replace` | Overwrite a file with the version from the zip |
| `patch` | Apply a unified diff to an existing file |
| `json_patch` | Apply structured add/remove/replace ops to a JSON or Python file |
| `mkdir` | Create a directory (skipped if already exists) |
| `run_sql` | Run a SQL statement against netwatch.db (safe to re-run — duplicate column errors are swallowed) |
| `restart` | Queue service restarts; `"services"` accepts `"web"`, `"monitor"`, or both |

For most updates, only `replace` and `restart` are needed.

---

### What patcher.py does automatically

- Takes a **full snapshot** of all files + database **before touching anything**
  (saved to `~/netwatch/snapshots/`, last 5 kept)
- Creates `.bak` copies of each file it overwrites
- Runs Python **syntax check** on all `.py` files before replacing them
- Updates the `VERSION` file on success
- Logs every step to the `patch_log` table in the database (visible in Admin → Changelog)
- If any step fails, remaining steps are **skipped** (not rolled back automatically —
  use Admin → Emergency Rollback or `bash ~/netwatch/rollback.sh` to recover)

**When to use:** Replacing one file when you already know exactly what changed.

**Limitation:** No version bump, no snapshot, no changelog entry. Use Method A for anything

## Auth — Password Reset (v3.2.69/70 behaviour, confirmed)

- `request_password_reset()` writes temp hash to `reset_token_hash` only — `password_hash` is never overwritten
- `authenticate()` checks both `password_hash` AND `reset_token_hash` — either works
- Both temp password and real password work simultaneously by design
- v3.2.75 incorrectly "fixed" this (breaking it), v3.2.76/77 repaired the damage — v3.2.77 is equivalent to v3.2.70

## Common Gotchas (learned the hard way)

- **No `get_setting()`** — use `getattr(config, "KEY", default)`.
- **No `@login_required` or `@require_role()`** — use `@auth.login_required` and `@auth.requires_permission("...")`.
- **`auth.get_current_user()` returns a dict** — access with `["id"]` not `.id`.
- **`sqlite3.execute()` is single-statement only** — never put two SQL statements in one `c.execute()` call.
- **`autoSkip:false` in Chart.js** compresses the x-axis — use `autoSkip:true` + `maxTicksLimit`.
- **`display:none` twice in an inline style** — second one wins; JS `.style.display = 'flex'` won't override it.
- **Pi runs in a venv** — `/home/mboynton/netwatch/venv/bin/python3`.
- **webapp.py has no `app.run()`** under gunicorn — `wsgi.py` is the entry point.
- **Read all transcripts before proposing fixes** — an hour was wasted "fixing" auth that was already working (v3.2.75 incident).
- **24H data gap after monitor restart is real, not a query bug** — the monitor simply wasn't writing. Don't chase the DB query.
- **`sample()` is gone from metrics.html** — replaced by `bucketAvg()`. Do not reintroduce it.
- **Package zip must be built from inside the package directory** — `cd /package_dir && zip -r output.zip .` — never zip from outside or files land in a subdirectory and the manifest won't be found.
- **`_resolve_path()` only auto-places files it recognises** — `.html` → `templates/`, `.css/.js` → `static/`, `dev_docs/` prefix → `dev_docs/` (v3.2.90+). All other subdirectories must use explicit full paths in the manifest.

## mBase Strategy — Read This Before Any Architecture Discussion

### What mBase Is and Why It Exists

NetWatch was built iteratively from scratch without prior knowledge of networked
application best practices. Many things were hand-rolled that could have been
borrowed, and some architectural choices were made by default rather than by
design. This was not wasted effort — it produced a working production system and
a deep understanding of what a networked app actually needs. But it means NetWatch
carries some technical debt and some deviations from standards.

The user has ~15 future project ideas. The goal of mBase is to build the hard,
reusable infrastructure once — auth, security, config, package management, backup,
alerting, the module system — so every future project starts from a solid, proven,
standards-aligned foundation instead of from scratch.

**mBase = NetWatch minus the network monitoring application layer.**
After extraction, NetWatch = mBase + network monitoring. When mBase gets improved,
NetWatch pulls it in. All future projects start from mBase.

### The Strategy (decided Session 15)

Do NOT start mBase from scratch. Continue developing NetWatch, correcting and
reworking things toward the standards as you go. By the time mBase extraction
begins, NetWatch will already contain the corrected versions of everything. Then
strip the NetWatch-specific pieces out to create mBase v1. Both projects end up
better simultaneously.

This means: when a PENDING DISCUSSION item in Audit_Findings.txt gets decided
and requires a code change, that change goes into NetWatch first. mBase inherits
the corrected version at extraction time.

https://gitea-pi.home:3000/mboynton/netwatch.git. The Gitea Pi (192.168.100.230)

**What failed:** Sessions 11-13 attempted to use Gitea as the session-start
source — Claude fetching context docs directly from Gitea via Claude in Chrome
instead of the user uploading a zip. This was repeatedly blocked by browser
extension security filters (session keyword filter, JWT token filter on dotted
filenames) and sessions were shut down. The infrastructure works; the automated
session-start workflow does not. Session 13 rolled back to the zip upload method.

**Three open discussions needed (not blocking current work, but important):**

1. **Version control and change tracking:** How to use git properly for NetWatch
   and mBase going forward — commit discipline, branching, what belongs in git
   (code + dev_docs? or code only?). The zip package workflow is fine for
   *deploying* to the Pi; git solves *version control and change history*.
   These are two different problems and both need to be solved correctly.

2. **Sharing and publishing:** What open source or commercial distribution looks
   like for these projects. GitHub vs self-hosted Gitea. Licensing decisions.
   README and documentation requirements. How mBase gets versioned independently
   from projects built on it.

3. **Session workflow improvement:** Whether there is a better way to get Claude
   context at session start that doesn't require the manual zip upload. The Gitea
   direct-fetch approach is the right idea but the Claude in Chrome method hit
   too many security filters. Other approaches (GitHub public repo + web_fetch,
   different automation method) have not been tried. Not a blocker; the zip
   workflow is reliable. Worth solving eventually for the multi-repo future.

**Do not attempt the Gitea Claude-in-Chrome session-start workflow again without
a new plan that addresses the filter problems documented in Gotchas.txt.**
