- Core mBase design philosophy established (Session 15):
  "mBase CONNECTS. It does not DUPLICATE."
  Before building any mBase component, the question is always: does a better version of
  this already exist that mBase should connect to instead? Things get BUILT only when
  nothing adequate exists for the deployment model. Everything else gets CONNECTED TO.
  This principle keeps mBase small, maintainable, and better than hand-rolled alternatives.

- DECISION: API versioning (Finding 11.1)
  Both NetWatch AND mBase use /api/v1/ path versioning from this point forward.
  NetWatch gets retrofitted in a future package (all template JS calls updated).
  mBase starts with it from the first line of code.
  Path versioning chosen over header versioning: simpler, visible, debuggable,
  correct for browser clients and community integrations.
  Breaking vs non-breaking change distinction documented. API documentation (OpenAPI)
  is part of definition of done for public API surface.

- DECISION: Logging format (Findings 7.1/7.2)
  Both NetWatch and mBase use structured JSON logging via structlog library.
  LOG_FORMAT config key: "pretty" (default, human-readable) or "json" (for aggregation).
  Central collection stack: Loki + Prometheus + Grafana on Gitea Pi.
  Promtail on each Pi ships logs to Loki over HTTPS (LAN CA handles certs).
  Log collection configurable at first-run wizard — local-only default, central optional.
  Endpoint is a URL, not assumed LAN-only; external/cross-WAN shipping supported.
  Not locked to Loki specifically — OTEL-compatible endpoint design means any backend works.
  Correlation IDs added: every request gets a short ID on all related log lines.
  Log level discipline: ERROR=broken, WARNING=unexpected/handled, INFO=normal, DEBUG=dev.
  Audit logs and application logs are separate streams with separate retention.
  Never log passwords, full auth tokens, or session IDs — non-negotiable rule.
  Browser-based log viewer in Admin UI scoped to handle JSON format.
  SSH + journalctl remains authoritative crash debugging path.
  jq installed on Pi as standard SSH tooling.

- SCOPE CHANGE: Host monitoring and device telemetry removed from NetWatch/mBase build scope.
  Discovery: Grafana + Node Exporter + Prometheus handles this better than anything that
  would be built custom. NetWatch was building visibility features (latency charts, speed
  test history, future host monitoring) because it was the only dashboard available — not
  because it was the right place for them. Grafana is the right place.
  NetWatch's true identity is the AUTOMATION layer, not the visibility layer:
    NetWatch owns: connectivity failure detection, automated recovery, GPIO/relay hardware,
    reset sequences, NetWatch-specific alerting, automation configuration UI.
    Grafana owns: historical latency/packet loss visualization, speed test trends,
    host metrics across all servers, cross-server pattern analysis.
  NetWatch dashboard charts (latency, packet loss, speedtest) to be reviewed once
  Grafana is running — some may be retired.
  Kiosk display scope narrowed: operational quick-glance NetWatch status only.
  Full infrastructure dashboard = Grafana on a screen.
